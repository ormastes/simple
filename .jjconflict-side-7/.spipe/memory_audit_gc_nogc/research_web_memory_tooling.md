# Web Research: Memory-Inspection/Verification Tooling for Simple

**Date:** 2026-06-11
**Question:** What memory-inspection/verification tooling should a young systems language adopt, such that it can be toggled ON/OFF with zero overhead when off (compile-time and runtime), and what is the cost when always-on?
**Context:** Simple has per-directory memory-model trees (`gc_async_mut` vs `nogc_*`), an interpreter mode + native (Cranelift/LLVM) mode, a mimalloc-based C runtime, and an existing compile-time alloc_checker pass.

## 2. On/Off Mechanism Comparison

Ranked by when-off cost (best first):

1. **Compile-time enforcement (D `@nogc`, Rust borrow checker, Nim ARC injection).** Zero runtime cost ever; the check exists only in the compiler. D's `@nogc` is transitive and template-inferred, so library code is checked once and the guarantee composes ([dlang.org/spec/garbage.html](https://dlang.org/spec/garbage.html)). This is the only category that is also zero-cost *when on*.

2. **Compile-time elision / separate build flavor (`cfg`/comptime/#ifdef; Go `-race`, all clang sanitizers, mimalloc MI_DEBUG/MI_SECURE builds).** The tooling code is not in the release binary at all — truly zero when off. Cost: binary/flavor proliferation, and bugs only appear in the instrumented flavor. All clang sanitizers and Go's race detector use this model ([Clang ASan](https://clang.llvm.org/docs/AddressSanitizer.html), [Go race detector](https://go.dev/doc/articles/race_detector)).

3. **Allocator-value / vtable swap at startup (Zig allocator parameter, Odin context.allocator, mimalloc's pluggable build).** One indirect call per allocation either way; swapping in a tracking allocator costs nothing extra when the plain allocator is installed. Zig's allocator-as-parameter and Odin's `context.allocator` make this idiomatic — the debug allocator records per-alloc stack traces and reports leaks at shutdown, and release builds simply pass a different allocator ([zig.guide](https://zig.guide/standard-library/allocators/), [pkg.odin-lang.org/core/mem](https://pkg.odin-lang.org/core/mem/)). Near-zero when off (the indirection already exists), zero extra code in hot paths.

4. **Link-time interposition (weak symbols, LD_PRELOAD, heaptrack attach).** malloc/free defined as weak/interposable symbols lets an external tool wrap them without rebuilding; zero overhead when no interposer is loaded ([interposition guide](https://nvankempen.com/2025/06/04/guide-using-dynamic-loader-interpose-malloc.html), [tcmalloc weak-symbol issue](https://github.com/gperftools/gperftools/issues/130)). This is how heaptrack achieves zero-cost-when-off ([heaptrack](https://github.com/KDE/heaptrack)).

5. **Sampling behind a runtime flag (GWP-ASan, tcmalloc sampled profiling, mimalloc MI_GUARDED).** Code ships in the production binary; when "off" or at high sampling interval the cost is one predictable branch per allocation — negligible but not literally zero. Chrome/Android run GWP-ASan fleet-wide on this basis ([arXiv:2311.09394](https://arxiv.org/abs/2311.09394)); tcmalloc sampled profiling is ~1% at 512 KB intervals ([PingCAP measurements](https://dzone.com/articles/troubleshooting-memory-leaks-deep-dive-into-common-heap-profilers)).

6. **External tools, no integration (Valgrind, Miri-style interpretation).** Zero when off by definition, but 20–30x (Valgrind) / 10–100x (Miri) when on — CI/dev only ([Valgrind manual](https://valgrind.org/docs/manual/valgrind_manual.pdf), [Microsoft Rust book ch.5](https://microsoft.github.io/RustTraining/engineering-book/ch05-miri-valgrind-and-sanitizers-verifying-u.html)).

**Build-flavor vs runtime-flag trade-off:** flavors give literal zero when off but require rebuilds and double the test matrix; runtime flags give instant toggling and production telemetry but leave a per-alloc branch and dead code resident. The industry pattern is *both*: heavy checkers as flavors (ASan/TSan in CI), sampling checkers as always-shipped runtime flags (GWP-ASan in production).

## 3. Fit for Simple (gc/nogc trees, interpreter+native, mimalloc, alloc_checker)

- The per-directory memory-model split (`gc_async_mut` vs `nogc_*`) is structurally the same guarantee D expresses with `@nogc` — but Simple enforces it by directory convention plus the alloc_checker pass. Making the no-GC guarantee a transitive, compiler-checked effect (D-style) costs zero at runtime and formalizes what the tree layout already promises.
- Simple already ships mimalloc, which natively provides all three tiers: `MI_DEBUG_FULL` (dev invariant checking), `MI_SECURE` (~10%, hardened), and `MI_GUARDED` (GWP-ASan-style sampled guard pages with `MIMALLOC_GUARDED_SAMPLE_RATE`, default 1/4000) — no new dependency needed ([mimalloc modes](https://microsoft.github.io/mimalloc/modes.html), [mimalloc readme](https://github.com/microsoft/mimalloc/blob/main/readme.md)).
- Interpreter mode is Simple's natural "Miri": the interpreter can carry precise per-allocation provenance/leak checking that would be too slow natively (Miri's 10–100x is acceptable there because the interpreter is already the slow path). Native mode should rely on compile-time elision + allocator swap.
- The C runtime boundary (rt_* externs) means clang sanitizers (ASan/LSan/UBSan) apply directly to the runtime in a CI build flavor with zero release cost.

## 4. Top-3 Recommendation for Simple

1. **(a) Compile-time `@nogc`-style effect enforcement (D model) — extend alloc_checker.** Make "this function/module performs no GC allocation" a transitive compiler-checked property tied to the `nogc_*` trees, with inference for generics like D templates. Overhead: **0 runtime on, 0 off** ([dlang.org/spec/garbage.html](https://dlang.org/spec/garbage.html)). Highest leverage: it turns the existing directory convention into a guarantee.

2. **(b) Debug tracking allocator behind a compile-time flag (Zig DebugAllocator / Odin Tracking_Allocator model), wired through mimalloc `MI_DEBUG_FULL`.** A `--debug-alloc` build flavor where the runtime allocator records alloc-site info and reports leaks/double-frees at exit (and `bin/simple test` uses it by default, like `std.testing.allocator`). Cost when on: per-alloc metadata + trace capture, fine for tests; **literal zero when off** because the release flavor never contains the code ([zig.guide](https://zig.guide/standard-library/allocators/), [mimalloc modes](https://microsoft.github.io/mimalloc/modes.html)).

3. **(c) GWP-ASan-style sampling in production via mimalloc `MI_GUARDED`.** Ship release binaries with guarded sampling available, tuned by `MIMALLOC_GUARDED_SAMPLE_RATE` (env-toggleable, default off or 1/4000). Catches real-world heap-buffer-overflow/UAF at **negligible overhead**, proven fleet-wide in Chrome/Android ([arXiv:2311.09394](https://arxiv.org/abs/2311.09394), [Chromium](https://www.chromium.org/Home/chromium-security/articles/gwp-asan/)). When-off cost: one sampled branch per alloc (~0%, not literal zero) — acceptable for production telemetry; builds without `-DMI_GUARDED` remain literal zero.

**Supplementary (CI only, no language work):** ASan+LSan flavor of the C runtime and compiled stage tests (~2x CPU / 2–4x RAM, [Clang docs](https://clang.llvm.org/docs/AddressSanitizer.html)); Valgrind memcheck spot-checks (20–30x, [Valgrind manual](https://valgrind.org/docs/manual/valgrind_manual.pdf)); interpreter-mode provenance checks as Simple's Miri analogue. Avoid: TSan/MSan as default gates (5–15x / 3x, whole-program constraints), bytehound-style full-trace profiling in anything long-running ([PingCAP](https://dzone.com/articles/troubleshooting-memory-leaks-deep-dive-into-common-heap-profilers)).
