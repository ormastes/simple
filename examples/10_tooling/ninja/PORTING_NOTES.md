# simple_ninja -- Porting Notes

## Subset Coverage

This implementation covers the core Ninja build file format:

### Supported
- `rule NAME` with `command`, `description`, `depfile`, `deps`, `pool`
- `build OUTPUT: RULE INPUT...` with all dependency types
- Implicit dependencies (`| dep`)
- Order-only dependencies (`|| dep`)
- Variable declarations and `$var` / `${var}` expansion
- Per-build variable overrides (indented under build edges)
- Built-in variables: `$in`, `$out`, `$in_newline`
- `pool NAME` with `depth = N`
- `default TARGET...`
- `include FILE`
- `subninja FILE`
- Dependency-based rebuild via mtime comparison
- `-j N` parallel execution
- `-n` dry-run mode
- `-t clean` to remove outputs
- Line continuations with trailing `$`

### Not Supported (needed for real-world builds)
- **Dependency log** (`build.ninja.d`, `.ninja_deps`, `.ninja_log`):
  Real Ninja maintains a binary dependency database for fast incremental
  builds. This implementation re-scans mtimes each invocation.
- **depfile parsing**: gcc-style `.d` dependency files (`-MMD -MF $out.d`)
  are referenced but not parsed. Header changes won't trigger rebuilds
  unless listed as explicit or implicit deps.
- **Response files**: `rspfile` and `rspfile_content` are parsed but not
  actually written (needed for Windows command-line length limits).
- **Ninja re-generation**: `build build.ninja: GENERATOR_RULE` convention
  where Ninja re-runs itself if the build file changes.
- **Multiple outputs per edge** with `|` (implicit outputs) syntax.
- **Console pool**: the built-in `console` pool that attaches to the
  terminal for interactive commands.
- **Escape handling**: `$:` (literal colon), `$$` is handled but edge cases
  in path escaping may differ.
- **Stat cache**: real Ninja caches stat results within a build to avoid
  redundant filesystem calls.
- **Build manifest hashing**: real Ninja tracks command changes (if the
  compile flags change, it rebuilds even if files are untouched).

## Building Real-World Projects (like LLVM)

LLVM's Ninja build (generated by CMake) typically has 5000+ build edges.
To handle that scale:

1. **Dependency log** -- without `.ninja_deps`, every build does a full
   mtime scan. For 5000+ edges this adds noticeable overhead.

2. **depfile support** -- LLVM compilation uses `-MMD` to generate header
   dependency files. Without parsing these, header changes won't trigger
   the right recompilations.

3. **Response files** -- on Windows, LLVM link commands exceed the 32K
   command-line limit. Response files write arguments to a temp file.

4. **Memory efficiency** -- the current implementation stores all build
   edges in memory. For very large builds, streaming or lazy loading of
   the manifest would help.

5. **True parallel execution** -- the current parallel mode runs batches
   sequentially. A proper job queue with process spawning and completion
   polling would better utilize CPU cores.

## Cross-Compilation Considerations for SimpleOS

- SimpleOS processes are single-threaded by default. Parallel execution
  (`-j N`) relies on `process.run` spawning separate OS processes.
- For cross-compilation, the `cc` variable in build.ninja should point
  to the cross-compiler (e.g., `clang --target=x86_64-simpleos`).
- SimpleOS filesystem may not support the same mtime granularity as
  Linux/macOS. Build correctness depends on mtime resolution.
- The `sh -c` command execution assumes a POSIX shell. On bare SimpleOS,
  a minimal shell or direct process execution would be needed.
- Integration with `simple_cmake` for the configure step: run
  `simple_cmake -G Ninja` to produce `build.ninja`, then `simple_ninja`
  to execute it.
