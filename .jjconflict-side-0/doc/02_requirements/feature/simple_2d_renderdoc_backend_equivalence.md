# Simple 2D RenderDoc Backend Equivalence Requirements

Selected by the user on 2026-07-10: **Feature Option C — Full Cross-Backend Intensive Matrix**.

## Functional Requirements

- REQ-001: Define a versioned `BackendRenderRecord` that records requested/actual backend, adapter/device, native/translated/emulated execution, render target, viewport/scissor, pipeline and shader identities, resource bindings, ordered commands, synchronization/transitions, frame completion, readback provenance, dimensions, format, stride, and deterministic hashes.
- REQ-002: Reject malformed, incomplete, unsupported-version, contradictory-provenance, empty-frame, non-complete, and invalid-handle records with specific errors.
- REQ-003: Provide deterministic serialization plus `BackendRenderRecordDiff` with field paths, expected/actual values, first difference, and all differences; only a documented nondeterministic-field allowlist may normalize values.
- REQ-004: `BackendRenderEquivalenceResult` passes only when independently produced detailed records agree, each proves real completion and a positive backend handle, and nonblank readbacks match both an absolute fixture oracle and each other with zero mismatches and no tolerance.
- REQ-005: Route app/system rendering through `Engine2D.create_requested_backend` and the production web-render facade; backend-library tests may inspect owners directly, but app/spec code must not add raw runtime calls or silently construct implementations.
- REQ-006: Correctly distinguish requested backend, actual backend, native API, translation layer, readback source, and fallback reason. CPU or host-array pixels must never be labeled `device_readback`.
- REQ-007: On Linux, exercise real NVIDIA Vulkan and Mesa software Vulkan as distinct provenance lanes, plus explicitly labeled DirectX-request→Vulkan and Metal-request→Vulkan compatibility lanes where implementation evidence is real.
- REQ-008: Native Windows Direct3D 11/12 and native macOS Metal remain fail-closed external checkpoints with the same record/diff/oracle contract; Linux translation evidence cannot satisfy those checkpoints.
- REQ-009: Open and inspect RenderDoc captures through canonical helpers, proving replay-open success, API/device identity, relevant actions, pipeline/resources, capture/log paths, and real artifact validity; four-byte magic, screenshots, env files, or synthetic files alone fail.
- REQ-010: Replace simulated/placeholder RenderDoc specs with modern SSpec scenarios using `std.spec.*`, user-observable outcomes, `step("...")`, `@req`, manual sections, meaningful captures, and direct matchers.
- REQ-011: Cover Simple 2D primitives, alpha/blend, gradients, borders/radius/shadow, transforms, clipping/masks, text/glyphs, images, resources/transitions, resize/viewport, invalid inputs, deterministic replay, mismatch reporting, fallback rejection, and repeated frames.
- REQ-012: Cover all 50 production web-layout cases, the 132 deterministic famous-site corpus, and the 43-widget fixture through the production facade, asserting DOM/Draw IR/layout evidence before exact pixels and detailed backend equivalence. Text antialiasing divergence remains explicitly tracked, never memorized or tolerated into equality.
- REQ-013: Provide one canonical aggregate checker that reports every Linux/native-host/backend/corpus/stress gate, exact blockers, artifact paths, timing, max RSS, and requirement traceability.
- REQ-014: Generate readable operator manuals under `doc/06_spec`; executable specs stay under `test/`. Update architecture, design, test plan, agent plan, operator guides, workflow contracts, and reports affected by the new schema/checkers.
- REQ-015: Record cooperative sidecar findings, merge ownership, generated-manual review, and highest-capability final acceptance; sidecar-generated exclusions or done marks are not authoritative without that review.
- REQ-016: Harden the same Simple 2D facade for guest-native SimpleOS execution under QEMU and portable physical-board deployment. QEMU evidence must build and boot the target image, render inside the guest, emit a compact ordered backend-render receipt over serial, capture the presented framebuffer through QMP or the guest device, and match an independent exact oracle. The board contract must avoid host RenderDoc/process dependencies and record architecture, board identity, image/flash/boot path, renderer/backend, framebuffer or GPU handle, completion marker, capture/readback origin, dimensions/format/hash/nonblank count, and serial or SSH transcript; without physical-board evidence its status remains `qemu-verified` or `source-present`.
- REQ-017: QEMU capture and serial correlation use the existing pure-Simple QMP and OS harness facades. Remove inline Python capture/PPM/baseline helpers, automatic baseline creation, tolerance thresholds, and pass-on-backend-init-failure branches; exact guest evidence requires zero pixel mismatches.
- REQ-018: One `SimpleOsRenderTargetEvidence` schema covers x86_64 framebuffer/VirtIO, RV64 VirtIO/display, ARM64 framebuffer, QEMU, and physical boards, including guest run/frame IDs, architecture/SoC/display controller, firmware hash, boot transport, driver/scanout/resource IDs, DMA/cache/IOMMU mode, format/geometry/stride, handle, present sequence/fence/completion, guest receipt/readback, external capture, and oracle result.
- REQ-019: Physical-board PASS requires correlated real evidence from one boot: board identity/model/revision/serial hash, firmware/image hash, flash/download and reset/boot path, guest completion receipt, positive renderer/framebuffer/GPU handle, guest readback or honest unavailability reason, external framebuffer/device capture, exact oracle, and serial or SSH transcript. Static catalog entries or QEMU evidence cannot promote a board row.
- REQ-020: SimpleOS QEMU provides honest SIMD rendering lanes for x86/x86_64 (SSE2 and AVX2 where exposed), AArch64 (NEON), and RISC-V 64 (RVV with an explicitly vector-enabled QEMU CPU). Each lane records detected and required features, selected provider, vector width, actual native SIMD hit counts per operation, scalar-fallback counts/reasons, guest architecture, and exact framebuffer result; detection or provider selection without executed vector work cannot pass.
- REQ-021: The x86/x86_64 SimpleOS rendering gate executes real SIMD fill, copy/blit, alpha blend, and scroll or transform work inside QEMU, proves positive native SIMD hits and zero scalar fallback for required operations, matches an independently executed scalar oracle pixel-for-pixel, and confirms the built guest contains the expected SIMD instruction family rather than NOP runtime shims.

## Exclusions

- No claim that Linux provides native Windows DirectX or native Apple Metal evidence.
- No C/Rust rewrite used merely to bypass missing pure-Simple renderer behavior.
- No same-path equality, simulated counters, magic-only artifact checks, synthetic device handles, or fixture-specific painted references accepted as rendering proof.
- No host-side render substituted for the SimpleOS guest/board render, and no QMP fixed response or hardcoded serial marker accepted without a matching nonblank framebuffer oracle.
