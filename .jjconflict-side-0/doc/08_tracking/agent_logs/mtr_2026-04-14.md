# MTR Agent Log — 2026-04-14

## Task #40 — Matchers module missing boolean/type matcher exports

### Root cause
`src/compiler_rust/lib/std/src/spec/matchers/__init__.spl` used bare
`export be_true, be_false, ...` statements without the `from <submodule>`
clause. The symbols were defined in `boolean.spl` / `type.spl` etc., but
were never re-exported from the package `__init__`, so the interpreter
module loader flagged them as "Export statement references undefined
symbol".

### Fix
Rewrote `matchers/__init__.spl` to re-export every matcher function +
struct using the `export X from Y` pattern (the same form used in
`spec/__init__.spl`). Covers:
- `core` — `eq`, `be`, `be_nil` + matchers
- `boolean` — `be_true`, `be_false`, `be_truthy`, `be_falsy` + matchers
- `comparison` — `gt`, `lt`, `gte`, `lte` + matchers
- `collection` — `include`, `be_empty`, `have_length`, `have_size` + matchers
- `string` — `include_string`, `start_with`, `end_with`, `be_blank` + matchers
- `error` — `raise_error` + matcher
- `type` — `be_instance_of`, `be_a`, `be_an`, `be_option`, `be_result` + matchers

### Verification
`bin/simple test test/system/simpleos_desktop_disk_boot_spec.spl` and
`bin/simple test test/system/engine2d_in_qemu_spec.spl`: the
`undefined symbol name=be_*` and `Be*Matcher` warnings are gone. Specs
now progress past the matcher-import phase; remaining failures are
downstream (QEMU/environment-skips) and out of scope for task #40.

No new matcher code was needed — the matchers were already implemented
in their submodules; only the package-level re-exports were missing.
