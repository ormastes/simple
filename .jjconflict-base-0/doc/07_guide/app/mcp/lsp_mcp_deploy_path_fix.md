# Simple LSP MCP ‚Äî deploy-path hardening (2026-06-14)

## TL;DR

The `simple-lsp-mcp` server was **already working** when investigated ‚Äî live
`lsp_symbols` / `lsp_diagnostics` calls returned correct results and the
deployed binary was current. The real, recurring problem was in **setup**, not
the server: `.mcp.json` pinned a hand-maintained alias path that drifts out of
date on every rebuild. This change points the config at the canonical deploy
directory so the drift can no longer happen, and rebuilds/redeploys the binary.

## What was actually wrong

`bootstrap-from-scratch.sh` deploys MCP binaries to the **canonical platform
triple** directory:

```
bin/release/x86_64-unknown-linux-gnu/simple_lsp_mcp_server   <- deploy target
```

But `config/mcp/install.shs` generated `.mcp.json` with a hard-coded **legacy
alias** path:

```
bin/release/linux-x86_64/simple_lsp_mcp_server               <- launch path (.mcp.json)
```

In that alias directory, `simple` is a symlink to the canonical dir (so it
always tracked rebuilds), but `simple_lsp_mcp_server` was a **real file copy**.
Nothing in `setup.shs` recreates it ‚Äî the symlink loop there only covers
`t32_mcp_server` / `t32_lsp_mcp_server`, and the wrapper generation only covers
`simple_mcp_server`. So after every LSP rebuild the alias copy had to be
re-copied **by hand**, and whenever that was forgotten the LSP MCP launched a
stale binary. The hard-coded path was also not platform-aware (wrong on macOS).

This is the root cause behind the long-standing "LSP MCP stale deploy path"
note: it was never a server bug, it was a config that pointed somewhere the
build does not write.

## The fix

`config/mcp/install.shs` now resolves the LSP native binary from a candidate
list, **canonical deploy dir first**, using `scripts/setup/platform-detect.shs`
for the triple:

1. `bin/release/<platform-triple>/simple_lsp_mcp_server`       (canonical ‚Äî where bootstrap deploys)
2. `bin/release/<platform-triple>-macho/simple_lsp_mcp_server` (macOS variant)
3. `bin/release/linux-x86_64/simple_lsp_mcp_server`            (legacy alias, fallback)
4. `build/bootstrap/mcp-package/simple_lsp_mcp_server`         (package fallback)

Because the generated `.mcp.json` now launches the canonical path, future
`bootstrap-from-scratch.sh` redeploys are picked up automatically ‚Äî no manual
re-copy.

Secondary hardening: the stale real-file copy at
`bin/release/linux-x86_64/simple_lsp_mcp_server` was converted to a symlink into
the canonical dir, matching the `simple` symlink already in that directory, so
the legacy alias self-heals too. (That directory is gitignored, so this is a
local-state fix only.)

## Why I believe it is fixed

This is verification evidence, not a hypothesis about an unseen crash:

- **Setup root cause removed at the source.** `.mcp.json` is regenerated by the
  patched `install.shs` and now reads
  `‚Ä¶/bin/release/x86_64-unknown-linux-gnu/simple_lsp_mcp_server`, i.e. the exact
  directory `bootstrap-from-scratch.sh` writes to. The deploy target and the
  launch target are now the same path; the drift mechanism is gone.
- **Rebuilt binary passes the project's own smoke harness.** Against the
  deployed launch path, `scripts/check/check-mcp-native-smoke.shs` reports
  `lsp_tools_json_valid=true`, `lsp_framing_valid=true`,
  `lsp_tools_schema_valid=true`, `lsp_tools_count=11`, startup ‚âà6 ms.
- **Rebuilt binary is behaviorally identical to the previously-deployed,
  known-good one.** An identical hand-driven JSON-RPC `tools/call` produces the
  same response from both; the live MCP client separately exercised the real
  `lsp_symbols` path successfully. (The 132 stub functions emitted during the
  cranelift native-build are compiler-internal symbols pulled in by the
  `--source src/compiler` closure but never reached at runtime ‚Äî the LSP server
  is a thin JSON-RPC wrapper that **shells out** to `bin/simple` +
  `src/lib/nogc_sync_mut/lsp/lsp_query.spl` for the actual queries. The
  prior working binary carried the same stubs.)

## Scope / what was NOT verified

- Only the **MCP-bridge** path was exercised (the `simple-lsp-mcp` server used
  by Claude Code / Codex). Editor-side LSP integrations (VS Code / JetBrains)
  are a different surface and were not tested here.
- `lsp_hover` returns `null` for a plain function name. This is the lint-backed
  bridge having no hover payload for that symbol kind, not a regression ‚Äî it is
  unchanged from the prior binary.

## Operational notes

- Canonical deploy dir: `bin/release/<platform-triple>/` (Linux:
  `x86_64-unknown-linux-gnu`).
- Regenerate config after any rebuild: `sh config/mcp/install.shs`.
- The LSP server does the real work by spawning `bin/simple` with
  `src/lib/nogc_sync_mut/lsp/lsp_query.spl` (symbols/definition/references/‚Ä¶) and
  `simple lint` (diagnostics). Symbol resolution is **project-root relative**;
  pointing the tools at a path outside the repo returns empty results, not an
  error.
