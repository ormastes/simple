## Task Type
feature

## Refined Goal
Harden and optimize the pure Simple GUI rendering stack so Engine2D, web, WM, image, taskbar, command-bar, and vector-font rendering have bit-exact cross-backend evidence, fail-closed hardware diagnostics, and repeatable performance comparisons against Node.js, Electron, and GPU-optimized GTK baselines on host and QEMU where available.

## Acceptance Criteria
- AC-1: Engine2D Vulkan, CUDA, OpenCL, CPU-SIMD, and scalar paths have strict probe diagnostics, no silent fallback in strict mode, and bit-exact primitive readback comparisons against the CPU/software reference for available backends.
- AC-2: CUDA, OpenCL, and CPU-SIMD accelerated 2D operations have executable evidence for fill, copy, alpha, blit or scroll, including checksums or per-pixel comparisons and fail-closed unavailable reports when hardware/toolchains are absent.
- AC-3: Vector-font rendering has CUDA/OpenCL optimization coverage or typed unavailable diagnostics, plus deterministic glyph/layout bitmap evidence with exact checksums and no blur/tolerance normalization.
- AC-4: Pure Simple web rendering through the Simple 2D engine has bitmap-level Node.js comparison evidence, no blur/tolerance matching, and performance evidence showing the optimized Simple path is faster than the selected Node.js baseline or explicitly fails closed.
- AC-5: Simple GUI on the Simple web renderer has bit-level consistency evidence against Electron for the same scene, including image, text, taskbar, and command-bar content with exact bitmap/hash comparison and no blur normalization.
- AC-6: Host WM and QEMU WM rendering have bit-level consistency evidence for desktop, image rendering, taskbar, command bar, window chrome, and representative app content, with live capture used where the environment provides it and explicit unavailable diagnostics otherwise.
- AC-7: WM performance is compared with a GPU-optimized GTK baseline on both host and QEMU where available, using same-scene frame/text/render workloads, repeated trials, timeout-bounded runs, and documented pass/fail thresholds.
- AC-8: Performance optimization is not limited to reports: hot rendering paths used by the evidence have optimized implementations or dispatches, with counters proving optimized-path hits and fallback-path behavior.
- AC-9: Sanity checks cover no-blur policy, exact-difference reporting, deterministic output, unavailable-backend handling, timeout handling, generated artifact integrity, and absence of false pass claims.
- AC-10: Parallel work is coordinated through disjoint evidence lanes for GPU/Engine2D, web/Electron/Node, and WM/QEMU/GTK, and the SPipe state records completed evidence, remaining blockers, and hardware/runtime dependencies.
- AC-11: Generated GUI WASM hardening covers the real `examples/ui/hello_wasm_gui.spl` CLI path, with emitted artifact magic/version checks, required GUI exports/events/strings, and fail-closed diagnostics when the deployed compiler cannot emit the artifact.
- AC-12: Simple JS/WebEngine performance claims include a Bun baseline; Simple may pass only when exact bitmap output matches and measured Simple runtime is equal to or faster than Bun for the benchmarked scene, otherwise the evidence fails closed.

## Scope Exclusions
- Claiming completion from blurred, anti-aliased tolerance, perceptual, or visually similar comparisons.
- Claiming hardware-backed acceleration when only source generation, probe availability, or fail-closed diagnostics have been proven.
- Requiring unavailable hardware, display servers, QEMU guests, or platform toolchains to pass on this host without recording explicit unavailable diagnostics.
- Claiming Node.js/Bun parity for the whole Simple JS engine before the concrete API/conformance and benchmark matrix exists.
