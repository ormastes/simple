# Feature: pure_simple_ssh_https_servers

## Raw Request
$sp_dev impl pure simple ssh, https server in pure simple either alpha or beta and just releaee mode. let it works hosted(linux) and simple os too. do with spark involve pherallel agents. minimize rt unless os provide it and to access it. consider various host access to impl in pure simple with sffi and update sffi and rt skill on spipe.

## Task Type
feature

## Refined Goal
Implement hosted Linux and SimpleOS SSH and HTTPS server stacks whose protocol logic is written in Simple, with alpha/beta/release mode selection, minimal runtime usage limited to OS access through documented SFFI/RT boundaries, and updated SPipe/SFFI/RT guidance.

## Acceptance Criteria
- AC-1: SSH server protocol behavior for version exchange, KEX, authentication, encrypted packet handling, channel open, and command execution is implemented in `.spl` modules, with runtime calls limited to documented OS access boundaries such as socket I/O, time, entropy, process, or filesystem access.
- AC-2: HTTPS server protocol behavior for HTTP request parsing/routing plus TLS record, handshake, and encrypted application-data handling is implemented in `.spl` modules, with runtime calls limited to documented OS access boundaries and any unavoidable crypto/entropy primitives explicitly listed.
- AC-3: The implementation exposes an explicit mode selector with `alpha`, `beta`, and `release` behavior; `release` avoids dual execution and is the normal production path, while `alpha` and `beta` provide comparison/diagnostic behavior where a native/SFFI fallback still exists.
- AC-4: Hosted Linux and SimpleOS entrypoints both use the same Simple protocol modules and only differ at the host-access adapter layer.
- AC-5: SFFI wrappers for SSH/HTTPS/TLS/HTTP are updated so their public names and docs distinguish protocol logic from host-access adapters, and no production wrapper silently bypasses the Simple protocol stack.
- AC-6: SPipe specs cover hosted Linux and SimpleOS paths for SSH and HTTPS, including at least one successful request/session, one failed-auth or failed-handshake case, and one release-mode run.
- AC-7: Generated/manual spec docs under `doc/06_spec` are updated for every new or changed executable SSpec, with no executable `*_spec.spl` placed under `doc/06_spec`.
- AC-8: SFFI/RT process guidance is updated in the relevant SPipe skill/docs so future work knows that runtime usage is allowed only for OS access and must not replace Simple protocol behavior in release mode.
- AC-9: The work records Spark/parallel-agent usage attempts and outcomes; if Spark quota is unavailable, the state/log documents the fallback to main-agent or non-Spark work.
- AC-10: Verification includes focused checks for touched SSH/HTTPS modules plus `sh scripts/setup/install-spipe-dev-command.shs --check` and `find doc/06_spec -name '*_spec.spl' | wc -l` returning `0`.

## Scope Exclusions
- Do not require implementing TCP/IP, kernel scheduling, filesystem, entropy, or wall-clock primitives in pure Simple when the host OS or SimpleOS already provides them.
- Do not remove existing runtime/SFFI adapters until the Simple protocol path has equivalent verified coverage.
- Do not claim full TLS cipher-suite coverage unless the implemented suites have executable conformance evidence.
- Do not fold unrelated dirty GPU, browser, database, office, or live-gate recovery files into this lane.
