## Overview

    Verifies the modern Simple Web window-manager shell at the contract level.
    The shell remains server-authoritative, so this spec checks generated CSS,
    generated preview HTML, browser-side lifecycle hooks, and capture-audit
    tooling rather than issuing local state mutations.

    The user-facing goal is a modern desktop shell with Mac-like window
    lifecycle motion, rounded translucent surfaces, left-side traffic controls,
    desktop widgets, animated wallpaper, command/search titlebar input, modern
    round icon treatment, and an explicit way to reduce or disable verbose
    effects. The spec is intentionally broad because the WM UX is composed from
    CSS tokens, generated preview fixtures, JavaScript runtime behavior, and
    browser capture evidence.

    **Requirements:** doc/02_requirements/nfr/simple_wm_modernization_options.md

    **Plan:** doc/03_plan/os/wm/simple_gui_wm_restart_2026-05-28.md

    **Design:** doc/05_design/ui/misc/stitch_design_system.md

    **Research:** doc/01_research/ui/wm/simple_wm_modernization.md

    Related context also includes
    doc/01_research/ui/wm/simple_wm_modernization_local.md,
    doc/03_plan/ui/tui/harden_tui_gui_layout_comparison.md, and
    doc/05_design/os/desktop/shared_wm_renderer_unification.md.

    ## Syntax

    This is an executable SPipe specification. Each scenario uses
    `expect(value).to_contain(text)` for generated HTML/CSS/JavaScript
    contracts and focused numeric matchers such as `to_equal`,
    `to_be_greater_than`, and `to_be_less_than` for quality-report fields.
    The spec deliberately checks strings that must appear in the generated WM
    artifacts because the renderer builds the browser shell as HTML, CSS, and
    JavaScript source.

    ## Examples

    Example 1: generate the real WM page and verify theme CSS.

    `val html = generate_wm_html_page("glass_dark", "Simple WM", 8080)`

    The page must include rounded window chrome, translucent taskbar material,
    Mac-like open/close/minimize keyframes, focus rings, traffic-light controls,
    titlebar command input, animated wallpaper tokens, and reduced/off motion
    selectors.

    Example 2: generate the standalone preview and verify visible audit
    affordances.

    `val preview = generate_wm_modern_preview_html("glass_dark")`

    The preview must contain desktop widgets, command palette, control center,
    effect controls, quality inspector, browser-audit metadata, normalized
    round icons, and a taskbar preview so a reviewer can open one HTML file and
    inspect the target desktop state.

    Example 3: write the preview and produce browser-capture instructions.

    `write_wm_modern_preview_html("build/simple_wm_modern_preview.html", "glass_dark")`

    `wm_modern_preview_browser_audit_command(...)`

    The browser audit command points at the repo-native Electron capture tool.
    It captures ARGB pixels and a DOM audit JSON containing selectors,
    computed style, bounds, clipping, contrast, touch-target, and unexpected
    overlap pass/fail evidence.

    ## Scenario Guide

    1. Generated WM page CSS proves the theme layer: color tokens, glass
       material, radius tokens, scrollbar rounding, wallpaper animation,
       lifecycle keyframes, and off/reduced effect policy.
    2. Runtime JavaScript source proves the behavior layer: window creation,
       traffic buttons, icon normalization, command palette selection,
       titlebar input, context menu actions, widgets, and preference setters.
    3. Generated preview HTML proves the reviewer surface: all expected shell
       affordances are visible in one browser-loadable fixture.
    4. Quality report fields prove measured policy: contrast threshold,
       fixed dimensions, titlebar/taskbar sizing, touch target minimums,
       material policy, and reduced-motion support.
    5. Browser audit metadata proves how to validate rendered GUI quality:
       selectors, viewport, contrast minimum, touch minimum, clipping policy,
       unexpected-overlap policy, computed style, and screenshot readiness.

    ## Manual Review Checklist

    - Open `build/simple_wm_modern_preview.html` in a browser.
    - Confirm the desktop background is animated unless motion is reduced/off.
    - Confirm the top-left traffic controls look like close, minimize, and
      maximize buttons and remain accessible.
    - Confirm the titlebar shows an icon, title, context label, and command
      input suitable for browser/IDE workflows.
    - Confirm command palette, control center, effect controls, quality
      inspector, widget gallery, window overview, and taskbar preview are
      visible and bounded.
    - Confirm effect controls expose standard/reduced/off motion, standard/
      reduced/off transparency, and standard/subtle/off feedback.
    - Confirm rounded windows, widgets, taskbar, icons, controls, and
      scrollbars use the shared radius policy.
    - Confirm square icons are normalized into round modern icon surfaces.
    - Confirm the quality inspector reports contrast, sizing, motion,
      material, layout, icon, titlebar, and browser audit evidence.
    - Run the browser audit command from
      `wm_modern_preview_browser_audit_command` when Chromium/Electron
      dependencies are available.
    - Treat `audit_pass=false` as a GUI quality failure; inspect
      `failureReasons` for unexpected overlap, text clipping, contrast, or
      touch-target problems.

    ## Expected Evidence

    The generated page must include CSS tokens for `--ui-bg`, `--ui-text`,
    `--ui-window-alpha`, `--ui-command-alpha`, `--ui-taskbar-alpha`,
    `--ui-widget-alpha`, `--ui-corner-window-radius`, and motion duration
    variables. It must include selectors for `data-wm-motion=reduced`,
    `data-wm-motion=off`, `data-wm-transparency=reduced`, and
    `data-wm-transparency=off`.

    The generated preview must include semantic markers such as
    `data-preview="modern-wm"`, `wm-effect-controls`,
    `data-browser-audit="required"`, `data-effect-control="motion"`,
    `data-effect-control="transparency"`, and
    `data-effect-control="feedback"`.

    The browser audit JSON must include `pass`, `failureReasons`,
    `unexpectedOverlapCount`, `clippedCount`, `contrastFailures`,
    `touchFailures`, and the captured item list with bounding rectangles and
    computed style fields.

    ## Failure Triage

    If a string assertion fails, first determine whether the contract was
    intentionally renamed or whether the feature disappeared from the generated
    artifact. Rename-only changes should update this spec and the quality
    contract together. Missing generated artifacts indicate a regression in
    `src/app/ui.web/html.spl` or `src/app/ui.web/wm.js`.

    If the focused spec passes but browser review fails, inspect the generated
    preview HTML and the Electron audit JSON. A pass in this spec proves the
    shell contract and audit hooks exist; the browser audit proves the rendered
    pixels and computed DOM satisfy the policy on the target host.
