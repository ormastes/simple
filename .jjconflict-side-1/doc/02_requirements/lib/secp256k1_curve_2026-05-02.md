# Feature Request: secp256k1 Elliptic Curve — Pure-Simple Implementation

**ID:** FR-CRYPTO-SECP256K1-001  
**Date:** 2026-05-02  
**Requester:** BIP32 HD Wallet (blocked)  
**Priority:** P2

---

## Summary

BIP32 Hierarchical Deterministic wallet derivation requires secp256k1 elliptic
curve operations in pure-Simple. No such implementation exists in the repo.
`src/os/crypto/ecdsa_p256.spl` is FFI-only (delegates to `rt_ecdsa_p256_*`
externs); the runtime exposes **no secp256k1 externs**, so a pure-Simple math
implementation is required.

---

## Blockers for BIP32 (FR-CRYPTO-BIP32-001)

### Blocker 1 — secp256k1 curve operations (this FR)

Required surface:

| Function | Signature | Notes |
|---|---|---|
| `secp256k1_point_add` | `(P: [u8; 33], Q: [u8; 33]) -> [u8; 33]` | compressed SEC1 in/out |
| `secp256k1_scalar_mult` | `(k: [u8; 32], P: [u8; 33]) -> [u8; 33]` | compressed SEC1 out |
| `secp256k1_point_to_compressed` | `(x: [u8; 32], y: [u8; 32]) -> [u8; 33]` | 0x02/0x03 prefix |
| `secp256k1_generator_mult` | `(k: [u8; 32]) -> [u8; 33]` | k*G, returns compressed pubkey |
| `secp256k1_scalar_add_mod_n` | `(a: [u8; 32], b: [u8; 32]) -> Option<[u8; 32]>` | (a+b) mod n; None if result is 0 |
| `secp256k1_scalar_in_range` | `(k: [u8; 32]) -> bool` | k ∈ (0, n); n = secp256k1 order |

Constants needed:

- `SECP256K1_N: [u8; 32]` — curve order  
  `0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141`
- `SECP256K1_P: [u8; 32]` — field prime  
  `0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEFFFFFC2F`
- `SECP256K1_Gx`, `SECP256K1_Gy: [u8; 32]` — generator point coordinates

Implementation note: requires 256-bit big-integer arithmetic (add, subtract,
multiply, modular inverse). Can reuse patterns from `src/os/crypto/curve25519_bigint.spl`
but secp256k1 has a different field prime and group law.

### Blocker 2 — Base58Check encoding/decoding

BIP32 serialized keys use Base58Check (base58 + 4-byte SHA-256d checksum).
No `base58` module exists anywhere in `src/` or `src/lib/`.

Required surface:

| Function | Signature |
|---|---|
| `base58check_encode` | `(payload: [u8]) -> text` |
| `base58check_decode` | `(s: text) -> Result<[u8], Base58Error>` |

Suggested location: `src/lib/common/encoding/base58.spl`

## BIP32 Consumer Requirements (for implementer reference)

The BIP32 derivation path uses the above primitives as follows:

1. **Master key**: `HMAC-SHA-512("Bitcoin seed", seed)` — already exists in
   `src/lib/common/crypto/hmac.spl` as `hmac_sha512_bytes`.

2. **Child key derivation (CKD priv)**:
   - Non-hardened: `HMAC-SHA-512(parent_chain_code, secp256k1_generator_mult(parent_priv) || ser32(i))`
   - Hardened: `HMAC-SHA-512(parent_chain_code, 0x00 || parent_priv || ser32(i))`
   - `child_priv = secp256k1_scalar_add_mod_n(IL[0..32], parent_priv)` — None = invalid, skip

3. **Neuter (priv→pub)**: `secp256k1_generator_mult(priv_key)`

4. **Serialization**: 78-byte payload → `base58check_encode`

5. **Validation**: `secp256k1_scalar_in_range(master_priv)` after HMAC-SHA-512

---

## Acceptance Criteria

- [ ] All six functions above compile and pass unit tests in interpreter mode
- [ ] BIP32 Test Vector 1 master key derivation round-trips through `secp256k1_generator_mult`
- [ ] `secp256k1_scalar_add_mod_n` returns `None` when result == 0 (per BIP32 spec §child-key-derivation)
- [ ] `base58check_encode` + `base58check_decode` are inverses for arbitrary payloads
- [ ] No FFI wrappers — pure-Simple math only
