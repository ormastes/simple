# MCP Startup / Handshake Performance Research (2026-06-12)

External research pass on why `bin/simple_mcp_server` feels slow vs Python/Node MCP servers.
Companion plan: `doc/03_plan/app/mcp/mcp_startup_perf_small_tasks_2026-06-12.md`.

## Core finding

The MCP protocol handshake itself is NOT the slow part. MCP initialization is just
version/capability negotiation; `tools/list` is the first expensive discovery step because
it returns tool metadata/schemas. The slow path is Simple's full MCP server startup
architecture — not MCP, and not Simple-the-language.

Measured baseline (from `doc/07_guide/app/mcp/startup_performance.md`):

| Path | Measured startup |
|------|------------------|
| direct full MCP native candidate | ~1360 ms |
| wrapper cold / stale probe | ~2720 ms |
| wrapper warm cached stamp | ~1366 ms |
| Simple LSP MCP | ~50 ms |

Full server advertises 151 tools (38 KB `tools/list` payload); LSP MCP exposes 11 tools.

## Why it is slow

1. **Server size.** `src/app/mcp/main.spl` registers 151 tools and imports many "lazy"
   handler modules at the top level (JSON helpers, protocol, diagnostics, VCS, debug,
   dialog, query, play, assistant, editor, static tools, dispatcher, registry, table,
   CLI passthrough). Boot is not just "reply to initialize".
2. **Wrapper double-start.** The `bin/simple_mcp_server` wrapper (generated by
   `scripts/setup/setup.shs`) probes native candidates by launching them with a full
   `initialize` + `tools/list` sequence and grepping for `play_wm_text_status`. Cold/stale
   start = probe handshake + real exec ≈ 2720 ms; warm cached stamp skips it.
3. **Benchmark artifacts.** `test/03_system/tools/mcp/mcp_startup_test_system.shs` inserts
   `sleep 1` between messages (~3 s artificial). Use
   `scripts/check/check-mcp-native-smoke.shs` (no sleeps) for real numbers.
4. **Interpreter path cost.** Source fallback (`bin/simple run src/app/mcp/main.spl`) pays
   import+parse cost; native-vs-interpreted gap is expected 10–100×. Earlier dependency
   reduction moved closure 61 → 49 modules and module-load 130 ms → 72 ms.
5. **tools/list construction.** First `tools/list` calls `_mcp_static_tools_result()`,
   which builds the catalog by repeated string concatenation + schema generation. It is a
   real workload, not protocol overhead.
6. **npm/npx layer.** `npx @simple-lang/mcp-server` adds a package-manager launcher; prior
   report shows version drift between npm package, runtime, and bundled source. Ship
   native binaries in the bootstrap package and use them first.

## Highest-impact solutions (priority order)

1. Fix the benchmark: no-sleep harness; split timings into start→initialize,
   start→tools/list, start→first tools/call.
2. Compare direct native vs wrapper warm vs wrapper cold vs source fallback; treat source
   fallback as a deploy bug (keep `SIMPLE_MCP_ALLOW_SOURCE_FALLBACK=0` default).
3. Replace the wrapper's full-handshake probe with a cheap native self-check
   (`simple_mcp_server --probe`: version, build ID, tool-manifest hash, tool count).
   Keep the full handshake probe in CI/smoke only.
4. Generate the entire `tools/list` response as one pre-escaped literal at build time
   (`val MCP_TOOLS_LIST_RESPONSE = "..."`); `_mcp_tools_list_result()` returns it directly.
5. Make handler imports genuinely lazy: startup path = stdio framing + initialize metadata
   + generated manifest + tiny dispatcher table (tool_name → handler_kind/module);
   `tools/call` loads only the needed module. Interim if conditional imports are hard:
   category subprocesses (core / debug / play / assistant).
6. Reduce default tool surface: `tool_set = "core"` (~15–25 tools) default,
   `tool_set = "all"` (151) opt-in; MCP supports `tools/list` pagination/cursors.
7. Keep cutting import-graph cost with `bin/simple deps fast|normal|deep`; watch hub
   modules with `export use x.*`. First targets: `src/app/mcp/main_dispatch.spl`,
   `main_lazy_play_tools.spl`, `main_lazy_debug_tools.spl`, `main_lazy_assistant.spl`,
   `main_editor_tools.spl`, `src/lib/nogc_sync_mut/mcp_sdk/**`.
8. Avoid npx in perf-sensitive configs; benchmark absolute binary path vs
   `python server.py` / `node server.js`.
9. Keep-warm/daemon mode only as last resort (lifecycle complexity).

## Realistic target

Not "151-tool server as fast as a tiny Node/Python server". Target: default core MCP
behaves like `simple_lsp_mcp_server` (tens of ms); full 151-tool surface stays available
as an explicit, heavier server.
