# SimpleOS SMF App Runtime Plan

Date: 2026-04-20

## Ownership Assumptions

- File I/O is owned by another agent.
- Scheduler is owned by another agent.
- This plan owns IPC/window protocol, filesystem namespace contract, dynamic
  library loading, and SMF loader harmony.

## Recommendation

Make SMF the SimpleOS app and library container format.

Do not replace ELF as the first process-image format. Instead:

- `.smf` executable packages carry an embedded ELF executable plus app metadata.
- `.smf` library packages carry PIC code, imports, exports, relocations, init/fini
  hooks, ABI version, and dependency metadata.
- `.lsm` libraries remain indexed archives of multiple `.smf` modules.
- The kernel/app loader extracts the embedded ELF from executable SMF and passes
  it to the existing ELF loader.
- The dynamic linker consumes SMF library metadata and maps libraries into the
  process link namespace.

## Phase 1: SMF Loader Harmony

Goal: one canonical SMF parser behavior for compiler, userland, and kernel use.

Tasks:

- Fix `SmfReaderMemory.from_data` to detect v1.1 trailer headers at EOF-128 and
  v1.0/header-at-start files only as legacy fallback.
- Unify section type IDs between `smf_writer.spl`, `smf_reader_memory.spl`, and
  `smf_header.spl`.
- Add explicit section kinds for app manifest, import table, export table,
  dependency table, relocation table, ABI descriptor, and window hints.
- Treat `SMF_FLAG_EXECUTABLE` and `SMF_FLAG_PIC` as role-defining flags.
- Reject ambiguous files that are both executable and library unless a future
  multi-role flag is added.

Acceptance:

- A v1.1 SMF generated by `smf_writer.spl` loads through `SmfReaderMemory`.
- Embedded ELF extraction works from a trailer-header SMF.
- `.lsm` archive lookup returns the same SMF bytes as a single-file lookup.

## Phase 2: Executable SMF

Goal: apps can be shipped as SMF packages while still loading via the existing
ELF process loader.

SMF executable profile:

- `flags`: `EXECUTABLE`
- `code` section: full ELF executable bytes, not stripped text-only bytes
- `note.app`: app id, display name, entry path, ABI version
- `note.window`: initial window mode, title, size, surface requirements
- `note.ns`: requested namespace entries and capabilities
- `note.deps`: library dependencies by SONAME-like stable name and ABI version

Loader path:

- VFS resolves `/sys/apps/browser_demo.smf`.
- SMF reader validates magic, role, target arch, ABI version, and hashes.
- Loader extracts ELF from `code`.
- Existing ELF loader validates and builds the process image.
- Scheduler agent receives the runnable image.

Acceptance:

- `/sys/apps/browser_demo.smf` produces the same ELF bytes as current
  `/sys/apps/browser_demo` root ELF fallback.
- Bad SMF role, wrong arch, missing ELF, or ABI mismatch returns a typed error.

## Phase 3: SMF Dynamic Libraries

Goal: load shared SimpleOS libraries from `.smf` or `.lsm` without baking every
feature into each app executable.

Library profile:

- `flags`: `PIC`
- `exports`: symbol name, type, version, visibility, ABI hash
- `imports`: required symbols with library name and version range
- `reloc`: supported relocations only
- `init_array` and `fini_array`: optional startup/shutdown hooks
- `note.deps`: transitive dependencies

Runtime linker:

- Per-process link namespace with loaded-library table.
- Breadth-first dependency resolution for deterministic behavior.
- Reference counts for normal libraries.
- Isolated plugin namespace can come later, modeled after `dlmopen`.

First supported API:

- `dlopen(path, flags)`
- `dlsym(handle, symbol)`
- `dlclose(handle)`
- `dlerror()`

Acceptance:

- A process can load a PIC SMF library from `/usr/lib/simple`.
- Import resolution fails closed with a diagnostic if any symbol or ABI version
  is missing.
- Library init runs once per link namespace.

## Phase 4: Filesystem Namespace

Goal: apps see a normal host-OS-like filesystem view, while drivers remain
baremetal-aware.

Process namespace model:

- `root`: process root path
- `cwd`: current working directory
- `mount_view`: list of visible mount points
- `cap_dirs`: service directories such as `/svc`, `/dev`, `/run`
- `library_paths`: default `/lib`, `/usr/lib`, `/usr/lib/simple`

Rules:

- Apps use paths and file descriptors, never raw block devices.
- Drivers can access device and MMIO paths through kernel-owned namespaces.
- Dynamic linker searches only the process library paths plus explicit app
  manifest paths.
- `..` must not escape process root.

Acceptance:

- App path lookup resolves `/bin`, `/usr/bin`, `/sys/apps`, `/lib`, `/usr/lib`,
  `/tmp`, `/home`, and `/svc`.
- VFS fd table stores fd-to-mount ownership, not first-mount fallback.
- Per-process cwd changes affect only that process.

## Phase 5: IPC and Window Protocol

Goal: GUI apps talk to the WM/compositor as normal processes through IPC.

Protocol model:

- WM service is exposed at `/svc/window-manager`.
- App gets a service handle through namespace capability resolution.
- `WmCreateRequest`, input events, resize/move/close, focus, and surface-update
  messages use a stable binary ABI derived from `common.window_protocol`.
- Window ownership is bound to process id and app id.
- Process exit or crash closes owned windows.

Acceptance:

- No app calls resident WM functions directly.
- Window messages include process identity from kernel-owned metadata, not
  caller-provided trust-only fields.
- Existing desktop E2E shortcuts still pass after moving launch to IPC.

## Phase 6: Integration Order

1. SMF reader harmony.
2. Executable SMF wrapper around existing ELF app payloads.
3. Namespace metadata and library search path contract.
4. Dynamic SMF library loader with PIC-only support.
5. WM IPC service boundary.
6. Remove resident fallback once syscall/process launch is stable.

## Risks

- Trailer/header mismatch can make existing SMF readers reject new files.
- Dynamic linking before process isolation increases kernel attack surface.
- Window IPC must not trust app-supplied pid/app id fields.
- Library unloading is unsafe until no thread can execute code from the library.

## Open Dependency Hooks

- File I/O agent must provide reliable read/stat/readdir for `.smf`, `.lsm`, and
  library paths.
- Scheduler agent must provide process id, thread id, exit cleanup, and runnable
  image handoff.
- Process isolation work must provide per-process address spaces before dynamic
  libraries can be mapped safely.
