# SHB (Simple Header Binary)

SHB is a FlatBuffer-style binary format that caches the public interface of a Simple module. It stores only exported declarations â€” functions, structs, classes, enums, traits, re-exports, type layouts, and dependency hashes â€” so that consumers can inspect a module's API without parsing or loading its full source.

## Format Overview

An `.shb` file has a fixed-size header, a section offset table, and variable-length section data.

### Header (64 bytes)

| Field | Type | Description |
|-------|------|-------------|
| `magic` | `[u8; 4]` | `0x53 0x48 0x42 0x00` ("SHB\0") |
| `version_major` | `u8` | Format major version (currently 1) |
| `version_minor` | `u8` | Format minor version (currently 0) |
| `platform` | `u8` | Target platform (Linux, macOS, Windows) |
| `arch` | `u8` | Target architecture (x86_64, aarch64) |
| `source_hash` | `i64` | Hash of the source file content |
| `interface_hash` | `i64` | Hash of the public interface only |
| `write_timestamp` | `i64` | Microsecond Unix timestamp of generation |
| `section_count` | `i64` | Number of sections (fixed at 8) |
| `string_table_offset` | `i64` | Byte offset to string table |
| `string_table_size` | `i64` | String table length in bytes |
| `flags` | `i64` | Reserved flags |
| `module_name_offset` | `i64` | Offset into string table for module name |

All multi-byte integers are little-endian. String references are `u32` offsets into the string table.

Source: `src/compiler/80.driver/shb/shb_types.spl`

### Section Table (64 bytes)

Eight `i64` offsets, one per section index:

| Index | Constant | Section |
|-------|----------|---------|
| 0 | `SHB_SECTION_FUNCTIONS` | Public function signatures |
| 1 | `SHB_SECTION_STRUCTS` | Public struct definitions |
| 2 | `SHB_SECTION_CLASSES` | Public class definitions |
| 3 | `SHB_SECTION_TYPE_LAYOUTS` | Size/alignment for platform types |
| 4 | `SHB_SECTION_ENUMS` | Public enum definitions |
| 5 | `SHB_SECTION_TRAITS` | Public trait definitions |
| 6 | `SHB_SECTION_REEXPORTS` | Re-exported symbols (`use X; export X`) |
| 7 | `SHB_SECTION_DEPENDENCIES` | Dependency module paths + interface hashes |

Section data follows the table. `ShbReader` decodes sections on demand (lazy loading).

Source: `src/compiler/80.driver/shb/shb_reader.spl`

### Section Entry Types

| Entry Struct | Key Fields |
|-------------|------------|
| `ShbFnEntry` | name, params (`[ShbParam]`), return_type, flags |
| `ShbStructEntry` | name, fields (`[ShbFieldEntry]`), flags |
| `ShbClassEntry` | name, fields, methods (`[ShbMethodRef]`), flags |
| `ShbEnumEntry` | name, variants (`[text]`), flags |
| `ShbTraitEntry` | name, methods (`[text]`), flags |
| `ShbReexportEntry` | symbol_name, source_module |
| `ShbTypeLayoutEntry` | type_name, size, alignment, flags |
| `ShbDependencyEntry` | module_path, interface_hash |

## Generation Paths

SHB files are generated by extracting public declarations from a parsed AST. Two generation paths exist.

### External Path (production-safe)

`external_extract_write_shb(source_path, shb_path)` spawns a subprocess:

```
bin/release/simple src/app/dev/shb_emit.spl <source_path> <shb_path>
```

The emitter outputs a machine-parsable protocol on stdout:
- Success: `SHB_OK|<source_hash>|<interface_hash>` (exit 0)
- Failure: `SHB_ERR|<error_tag>|<message>` (exit 1), where error_tag is `read_failed`, `parse_failed`, or `write_failed`

After writing, the caller validates the file via `ShbReader.open()` + `is_valid()`.

This path is safe from any runtime context, including the interpreter. All production consumers (cache manager, watcher client) use this path exclusively.

Source: `src/compiler/80.driver/shb/shb_extractor.spl`, `src/app/dev/shb_emit.spl`

### In-Process Path (compiled binary only)

`parse_and_extract_shb(source_path)` and `parse_extract_write_shb(source_path, shb_path)` parse and extract in the same process.

**Not production-safe in interpreter runtime.** The frontend parser requires global init that is not re-entrant from the interpreter. Use only from compiled binaries (e.g., the `shb_emit` tool itself).

Source: `src/compiler/80.driver/shb/shb_extractor.spl`

## Consumers

### Cache Manager (`ShbCacheManager`)

Manages `.shb` files for incremental compilation with a three-level validation scheme:

1. **Level 1 â€” source hash:** Compare `source_hash` in the existing SHB against the current file hash. If unchanged, the source has not been modified. Delegates to `cache_validator.validate_source_hash()`.
2. **Level 2 â€” interface hash:** If the source changed, re-extract. Compare old and new `interface_hash`. If unchanged (`SHB_CACHE_BODY_ONLY`), dependents do not need recompilation. If changed (`SHB_CACHE_INTERFACE`), dependents are marked dirty.
3. **Level 3 â€” dependency hashes:** Validate each `ShbDependencyEntry.interface_hash` against the dependency's current SHB. If any dependency interface changed, re-extract even if this module's source is unchanged.

Cache result statuses: `SHB_CACHE_UNCHANGED` (0), `SHB_CACHE_BODY_ONLY` (1), `SHB_CACHE_INTERFACE` (2), `SHB_CACHE_NEW` (3), `SHB_CACHE_ERROR` (-1).

Source: `src/compiler/80.driver/shb/shb_cache.spl`

### Watcher Client

The watcher client (`get_or_generate_shb`) follows a three-step strategy:

1. **Freshness check** â€” call `check_shb_freshness()`. If `CACHE_FRESH`, open the existing SHB and return it.
2. **Daemon request** â€” if a watcher daemon is running (`is_watcher_running`), submit a generation request via `daemon_sdk.client` and poll for completion.
3. **External fallback** â€” if no daemon, spawn a subprocess via `external_extract_write_shb`.

The watcher client is policy-free: it provides freshness primitives and get-or-generate flows. Policy decisions belong to the loader and interpreter.

Source: `src/compiler/80.driver/watcher/watcher_client.spl`

### Loader (Compiled Module Loader)

The loader uses SHB as a preflight check before committing to a full SMF body load:

- Calls `check_shb_freshness()` and `ShbReader` to validate artifacts.
- Checks dependency interface hashes via `ShbDependencyEntry` to detect interface drift early.
- Goal: avoid loading a stale SMF.

Source: `src/compiler/99.loader/loader/module_loader.spl`

### Interpreter (Deferred Module Loading)

The interpreter uses SHB to gate lazy imports (`use lazy mod.*`):

1. Register deferred module metadata via `register_deferred_module`.
2. On symbol miss during evaluation:
   - Check explicit import list first.
   - Consult SHB symbol membership second (does the module export this symbol?).
   - Skip modules that cannot define the symbol.
   - Force-load only plausible modules via `force_deferred_module`.
3. After force-load, symbol table resolution proceeds normally.

This filtering prevents wildcard imports from forcing unrelated modules.

Source: `src/compiler/10.frontend/core/interpreter/module_loader.spl`

## Circular Dependency Model

Mutually importing modules (e.g., `A <-> B`) can produce SHB interfaces without recursion collapse.

### Generation Stack

A path stack (`shb_generation_push` / `shb_generation_pop`) tracks modules currently being generated. If module `B` is requested while `A` is in-progress:

- If `B` already has a valid (last-known-fresh) SHB, consume it.
- If not, allow interface-only extraction without body execution semantics.
- The stack prevents infinite regeneration recursion.

Source: `src/compiler/80.driver/shb/shb_extractor.spl`

### Generation States

Each module tracks an explicit generation state (`shb_extractor.spl:83-87`):

| State | Constant | Meaning |
|-------|----------|---------|
| `"missing"` | `SHB_STATE_MISSING` | No SHB exists on disk |
| `"generating-interface"` | `SHB_STATE_GENERATING` | Interface extraction in progress (on generation stack) |
| `"interface-ready"` | `SHB_STATE_READY` | SHB written and valid; interface hash available |
| `"stale"` | `SHB_STATE_STALE` | SHB exists but source or dependency hash mismatch |
| `"failed"` | `SHB_STATE_FAILED` | Generation attempted but failed |

When module `A` needs `B`'s interface and `B` is in `"generating-interface"` state, the system consumes the last-known-fresh SHB if one exists. This breaks cycles without requiring body evaluation.

### Interface-First Loading

The circular dependency model separates interface extraction from body-sensitive runtime loading:

1. SHB generation extracts only public declarations (no body execution).
2. Dependency interface hashes are validated after initial extraction.
3. Body loading is deferred until symbols are actually needed.

This allows two modules to each see the other's public API without requiring full body evaluation of either.

## Known Runtime Caveats

### Interpreter `.len()` Corruption

Module-level array `.len()` can corrupt the variable in interpreter mode. Workaround: use manual counters for module-level arrays. See `memory/bug_len_corruption.md`.

### Re-Entrancy Limitations

The in-process generation path (`parse_and_extract_shb`) is not safe from interpreter context because the frontend parser global state is not re-entrant. All interpreter-context generation must use the external subprocess path.

### No Daemon Dependency

The interpreter does not depend on watcher daemon availability. If no daemon is running, the external fallback path handles generation directly.

## Troubleshooting

### Stale SHB

**Symptom:** Module changes are not picked up by dependents.

**Diagnosis:** The `source_hash` in the cached `.shb` no longer matches the file content, but the cache was not invalidated.

**Fix:** Delete the `.shb` file from the cache directory (typically `build/headers/`). The next compilation or watcher request regenerates it.

### Invalid SHB

**Symptom:** `ShbReader.is_valid()` returns false, or error message `"invalid SHB readback"`.

**Cause:** Corrupted write, version mismatch, or truncated file.

**Fix:** Delete the `.shb` file. If the problem recurs, check that `shb_emit.spl` completes without errors:

```bash
bin/release/simple src/app/dev/shb_emit.spl <source_path> <output_path>
```

Verify the output protocol line starts with `SHB_OK|`.

### Interpreter Fallback Behavior

When the interpreter cannot use the in-process path:

1. It calls `external_extract_write_shb`, which spawns `bin/release/simple` as a subprocess.
2. If the subprocess fails (missing binary, parse error), the SHB is marked as missing.
3. The module is loaded fully (non-lazy) as a fallback â€” no SHB filtering is applied.

This ensures compilation never fails due to SHB unavailability; it only loses the lazy-loading optimization.
