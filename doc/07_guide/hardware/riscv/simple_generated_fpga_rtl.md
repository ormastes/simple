# Simple-Generated RV32/RV64 FPGA RTL

Date: 2026-07-18
Status: **contract-not-ready**

This guide is the operator path for the selected compiler-first RV32/RV64
Linux FPGA work. It deliberately distinguishes materialized contract files from
a CPU generated by the Simple compiler.

## Current truth

The bundle generator can materialize dual-architecture directories, profiles,
sidecars, VHDL interfaces, SoC scaffolding, formal files, and future acceptance
metadata. The current CPU VHDL is emitted placeholder text with constant RVFI;
it is not an executable compiler-generated CPU. PMP/translation owners now
exist in Simple source, but Sv32/Sv39 are not connected to the production
fetch/load/store bus, and no generated-RTL or KV260 Linux login/`ls` transcript
exists.

Accordingly, current bundles report:

```text
readiness = "contract-not-ready"
reason = "placeholder-core-no-semantic-rvfi"
authoritative_rtl_provenance = "none"
pure_simple_authoritative_rtl = "false"
```

Generated CPU, smoke, Linux-handoff, and formal testbench execution terminates
with:

```text
GENERATED_RTL_NOT_IMPLEMENTED lane=rv32
GENERATED_RTL_NOT_IMPLEMENTED lane=rv64
```

This is intentional. It prevents the old time-zero PASS reports from becoming
Linux, MMU, formal, or product evidence.

## Required compiler

Use a current pure-Simple self-hosted CLI. A binary that prints the Rust
bootstrap warning is not allowed for normal generation, testing, or docgen.
The canonical VHDL command is:

```bash
bin/simple compile --backend=vhdl path/to/hardware_root.spl -o build/path/entity.vhd
```

Successful production generation must also create the backend source map
`entity.vhd.map.json`, identify the compiler binary/hash, and reject unsupported
hardware constructs without fallback.

At the time of this guide update, the workspace `bin/simple` resolves to a Rust
bootstrap. An isolated pure Stage3 compiler can build the current full CLI, but
the resulting link contains unresolved startup/path stubs and source-oriented
commands such as `spipe-docgen` crash with exit 139. An older isolated pure
release CLI reproduces the same crash. Fix and qualify the full-CLI runtime
before treating any source execution, testing, or doc generation as acceptance.

## Generate the contract bundle

Once a current pure-Simple CLI is available:

```bash
bin/simple run src/hardware/fpga_linux/generate_riscv_fpga_bundle.spl /tmp/simple_fpga_bundle
```

Optional metadata flags:

```bash
bin/simple run src/hardware/fpga_linux/generate_riscv_fpga_bundle.spl \
  --board=mlk_s02_100t \
  --product-level=lab-rtl \
  --profile=compiler-first \
  --rv32-luts=25000 --rv64-luts=45000 \
  --rv32-mhz=50 --rv64-mhz=50 \
  /tmp/simple_fpga_bundle
```

These flags describe a target contract. They do not promote readiness or prove
the budgets.

Inspect:

- `/tmp/simple_fpga_bundle/riscv_fpga_rtl_manifest.sdn`
- `/tmp/simple_fpga_bundle/board_linux_boot_products.sdn`
- `/tmp/simple_fpga_bundle/rv32/rtl/simple_rv32gc_core.debug.json`
- `/tmp/simple_fpga_bundle/rv64/rtl/simple_rv64gc_core.debug.json`
- per-lane CPU, formal harness, `.sby`, and formal `.sdn` files

The manifest uses `contract_file`, not `authoritative_file`. Sidecars have
`authoritative: false`, no source-map claim, zero observed semantic RVFI ports,
no runner success markers, and `linuxBootValidated: false`.

## Operator flow

### 1. Generate RV32 and RV64 RTL from Simple sources

Production acceptance requires tracked hardware roots in the existing RV32 and
RV64 core capsules, a current pure-Simple compiler, deterministic nonempty
clocked/bus VHDL, source maps, and changing semantic RVFI.

The existing generated RV32 source contains useful scalar `@hardware`
decode/control helpers, but it is generated text and is not yet the product
core. The existing RV32/RV64 behavioral core files currently lack a compiler-
supported whole-core hardware boundary. Do not rename the contract placeholder
as authoritative output.

### 2. Exercise Sv32 and Sv39 translation plus PMP protection

The production CPU must call RV32 `MmuState`/`mmu_*` and RV64
`MmuState64`/`mmu64_*` for instruction and data paths, then apply CPU-owned PMP
before issuing the physical bus request. Acceptance includes SATP,
SFENCE.VMA, TLB refill/invalidation, page/superpage permissions, precise page
faults, A/D policy, TOR/NA4/NAPOT, R/W/X, priority, and locked PMP entries.

Current standalone MMU modules and bare-metal PMP write plans are not CPU
integration evidence.

Current source status (2026-07-18): both XLEN lanes have explicit 16-entry PMP
match/access-gate and `pmpcfg`/`pmpaddr` CSR owners. RV64 additionally stores
PMP in `CoreState64`, executes the six Zicsr forms through the M/S/PMP owners,
synchronizes accepted SATP writes with Sv39 state, suppresses illegal CSR
writeback, and enters delegated illegal-instruction traps. This is a real core
state integration slice. RV64 also has a fixed 16-entry ASID/global/superpage
TLB and a one-outstanding-read Sv39 walker that enforces canonical addresses,
SUM/MXR, fault-on-clear A/D, aligned leaves, reserved PTE rules, and S-effective
PMP before page-table requests. `memory64_start/cycle` now composes that walker
with final physical PMP and a one-outstanding-request fetch/load/store frontend,
including no-bus-on-fault and precise alignment/access causes. This is still not
core memory-path acceptance: the old permissive `lsu64_access()` identity path
has been deleted, but the protected frontend is not clocked into core
fetch/data arbitration, and RV32 core privilege/PMP wiring is incomplete. RAM-backed
`mmu*_translate*_test_adapter()` functions remain unit adapters, not RTL
evidence.

The focused pure-Simple source check currently exits 139 before diagnostics;
see `build/test-artifacts/riscv-f1n3/rv64_core_pmp_csr_check.log`. Do not rerun
the Rust seed or treat the unexecuted VHDL/GHDL scenario as PASS.

### 3. Boot Linux to an interactive login on generated RTL

Build and hash OpenSBI/firmware, Linux, DT, and rootfs separately for RV32 and
RV64. QEMU first validates the media/platform contract. A generated-RTL
simulator must then boot the identical hashes and produce DUT-origin OpenSBI,
kernel, init, and `login:` bytes.

QEMU success is recorded as `qemu-media`; it never satisfies `rtl-sim`.

### 4. Run terminal login and list guest files

The RTL terminal driver must send credentials and `ls /`, observe the accepted
shell prompt and deterministic root entries, and preserve byte-direction and
timestamp evidence. Output-only UART, testbench-origin prompts, reordered
markers, or an injected completion string fail.

### 5. Program each FPGA image and capture board-origin evidence

Build and program RV32 and RV64 KV260 bitstreams independently. Retain the
board/part/JTAG identity, compiler/RTL/media/bitstream hashes, DRC/timing/
resource reports, programming log, and cold/warm interactive transcripts.

The current host sees the KV260 Xilinx FTDI devices but no external PL UART
adapter. Use an attached 3.3 V PMOD UART or prove a bidirectional PS/PL serial
bridge. ILA-only markers and zero-byte UART captures are diagnostic evidence,
not login/`ls` proof.

## Evidence planes

| Plane | Proves | Cannot prove |
|---|---|---|
| `qemu-media` | firmware/kernel/DT/rootfs and platform contract | generated RTL or FPGA CPU |
| `rtl-focused` | entity/program/MMU/PMP execution | Linux boot or board implementation |
| `formal` | named properties bound to emitted DUT | Linux/terminal/physical behavior |
| `rtl-linux` | generated RTL boots and interacts with pinned Linux | FPGA timing or physical I/O |
| `synthesis` | implemented netlist, DRC, timing, resources | software boot or terminal input |
| `physical-board` | programmed KV260 cold/warm login and `ls` | another XLEN or another bitstream |

Each plane reports `pass`, `fail`, or `blocked`; no plane inherits success from
another.

## Gates

The public dual-lane smoke command is currently expected to return nonzero and
show the diagnostic placeholder:

```bash
sh scripts/check/check-riscv-rtl-linux-smoke.shs --timeout=30
```

It may return zero only after both generated CPU lanes retire real instructions
and their runners require DUT-origin success evidence.

Formal readiness and strict proof are separate:

```bash
sh scripts/check/check-riscv-formal-dual-track.shs
sh scripts/check/check-riscv-rtl-sby-proof.shs
```

Current placeholder harnesses deliberately fail. A port manifest, generated
BYL, or Lean profile proof cannot replace non-vacuous RVFI/SymbiYosys proof of
the emitted CPU.

FPGA preflight and live execution remain separate:

```bash
SIMPLE_BINARY=bin/simple sh scripts/check/check-riscv-fpga-simpleos-preflight.shs --local-only
sh scripts/fpga/check_kv260_simple_riscv_fpga.shs rv32 --with-ghdl
sh scripts/fpga/check_kv260_simple_riscv_fpga.shs rv64 --with-ghdl
```

Do not run these with the Rust bootstrap. Missing bidirectional terminal, DDR,
product bitstream, or board-origin login stays blocked.

## Acceptance and troubleshooting

- `GENERATED_RTL_NOT_IMPLEMENTED`: expected until a canonical compiled CPU
  slice replaces the placeholder. Do not suppress or reinterpret it.
- Rust bootstrap warning: restore the current pure-Simple CLI; do not use the
  seed for normal tooling.
- Full CLI exits 139 after a successful Stage3 link: inspect unresolved startup
  stubs in the native-build log; `--version` alone does not qualify the CLI.
- If GDB reaches `rt_process_run_inherit`, do not retry normal delegation. The
  current self-exec fallback also times out in source interpretation; follow
  the tracked full-CLI bug rather than substituting the Rust seed.
- Empty/constant RVFI or missing assertions: fail the formal lane.
- `login:` seen only in QEMU: media oracle passed; RTL/FPGA remains incomplete.
- zero UART bytes with an ILA marker: routing/adapter blocker, not terminal pass.
- negative timing slack or DRC errors: physical implementation fails even if
  simulation passes.

Requirements, architecture, detailed design, system-test plan, and executable
acceptance use the slug `riscv32_riscv64_fpga_simpleos_production`. The mirrored
SPipe manual must be generated from the executable spec after current pure-
Simple docgen is restored; a hand-written substitute is not acceptable.
