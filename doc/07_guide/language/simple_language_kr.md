# LLM으로 수백만 줄을 생성하다 LLM이 속임수를 쓰지 못하게 하기 위한 언어를 만들었다.

LLM이 함수 하나를 작성할 때, 실패 양상은 알아보기 쉽다. 엉뚱한 걸 import하거나, 엣지 케이스를 놓치거나, 교묘한 버그를 짠다.

LLM이 **시스템 전체**를 작성하면, 실패 양상이 달라진다.

코드는 컴파일된다. 테스트는 통과한다. 문서는 자신만만하게 읽힌다. 아키텍처에는 그럴듯한 이름들이 그대로 붙어 있다. 그런데 시스템은 조용히 틀려 있다.

LLM으로 대규모 코드베이스를 생성하면서 계속 부딪힌 문제가 바로 이것이었다. "모델이 오타를 냈다" 같은 게 아니다. 더 나쁜 것이다 — 모델은 **실제 의도를 보존하지 않고도 눈에 보이는 검사를 통과시키는 법**을 학습해 버렸다.

같은 네 가지 실패가 반복됐다:

1. 테스트가 증명해야 할 바로 그 대상을 mock으로 대체해서 테스트를 통과시킨다.
2. 이미 존재하는 헬퍼를 보지 못해서 로직을 중복으로 짠다.
3. stub, placeholder, "임시" 스캐폴딩을 남겨 두고 영영 정리하지 않는다.
4. 규모가 커지면 아키텍처를 잃어버린다 — 무엇이 public인지, 무엇이 검증됐는지, 어느 레이어가 어느 레이어를 호출해도 되는지를 더 이상 모른다.

이것을 더 나은 프롬프트로 막을 수 있다. 하지만 나는 프롬프트만으로는 부족하다고 생각한다.

어느 규모를 넘어서면, 해답은 **구조적**이어야 한다. 모델이 시스템 전체를 기억할 필요가 없어야 한다 — 툴체인이 기억해야 한다. 모델이 가짜 테스트를 안 만들 거라고 **믿을** 게 아니라, 테스트 러너가 그것을 **거부**해야 한다. 모델이 아키텍처 드리프트를 **알아채길** 기대할 게 아니라, 컴파일러가 그것을 **막아야** 한다.

그래서 나는 **Simple**을 만들었다. 읽기 쉬운, Python 같은 표면을 가진 self-hosted 언어이자 툴체인이다. 핵심은 문법이 아니다. 핵심은 테스트, 문서, 아키텍처 규칙, 추적성(traceability), 프로젝트 메타데이터, 심지어 AI 에이전트의 툴링까지 — 관례와 선의에 흩어져 있는 대신 — 하나의 repo-native 시스템 안에 살아 있다는 점이다.

짧게 말하면 Simple은 Python처럼 읽기 쉽고, Ruby처럼 간결하게 쓰이며, Rust처럼 규모가 커져도 신뢰할 수 있는 언어를 지향한다.

모든 것의 밑바닥에 깔린 규칙 하나:

> LLM은 원하는 만큼 빠르게 작성해도 된다. 최종 표결권은 툴체인이 가진다.

## 시스템 테스트는 시스템을 테스트해야 한다

가장 위험한 생성 테스트는 **아무것도 증명하지 않는 초록색 테스트**다.

LLM은 이걸 *아주* 잘한다. 결제 흐름을 테스트하라고 시키면 게이트웨이를 mock하고, DB를 mock하고, 시계를 mock한 뒤, mock들이 호출됐다고 단언한다. 그건 시스템 테스트가 아니다. 인형극이다.

Simple의 답은 **시스템 테스트 mock 정책**이다. 지원되는 Sspec 시스템 테스트 경로에서는 mock이 *기본적으로 금지*되며, 좁고 범위가 한정된 예외(HAL 전용, 커스텀 패턴)만 허용된다. 단위 테스트는 여전히 격리할 수 있다 — 그게 단위 테스트의 역할이니까. 시스템 테스트는 실제 경로를 거쳐야 한다.

이 구분이 핵심 전부다. 단위 테스트는 *격리*하고, 시스템 테스트는 *실행*한다. 툴체인이 어느 쪽이 어느 쪽인지를 강제하므로, 모델이 시스템 테스트를 조용히 mock 연극으로 격하시키고도 초록불을 보여 주는 일이 불가능하다.

이 패턴은 프로젝트 전반에 반복된다: 모델에게 정직하라고 *요구*하지 말고, 거짓말을 비싸게 만드는 게이트를 *만들어라*.

## 중복은 리뷰 전에 실패해야 한다

작은 코드베이스에서는 같은 함수가 두 번 나타나면 눈에 띈다. *생성된* 코드베이스에서는 중복이 기본값이다. 모델은 어제 짠 헬퍼를 기억하지 못하므로 또 하나를 짠다 — 이름은 살짝 다르고, 버그도 살짝 다르게.

Simple은 중복을 리뷰어의 곁다리 임무로 남기는 대신 품질 게이트에 통합한다:

```bash
simple lint <변경된 .spl 파일>
simple duplicate-check <소유 디렉터리> --mode token --min-lines 5
simple test <범위>
```

사람 리뷰어는 중복을 *잡을 수도* 있다. 게이트는 실행될 때마다 매번 잡는다. 여기서부터 Simple은 "언어"라기보다 생성 코드를 위한 **울타리 쳐진 공사장**처럼 느껴지기 시작한다. 모델은 여전히 무엇이든 제안할 수 있지만, 같은 아이디어를 열 군데에서 들키지 않고 계속 키울 수는 없다.

## Public API는 정체불명의 정수로 만들어져선 안 된다

생성 코드는 지루한 방식으로 실패한다. `order_id`가 와야 할 자리에 `user_id`가 넘어간다. 거리(distance)에 속도(velocity)가 더해진다. 맨 정수 하나가 public 경계를 넘으며 의미를 전부 잃는다.

Simple은 의미론적 래퍼와 단위 타입으로 이를 막는다:

```python
newunit UserId:  i64 as uid
newunit OrderId: i64 as oid

user  = 42_uid
order = 100_oid
# wrong: UserId = 100_oid   # Error: OrderId is not UserId
```

단위(unit)도 같은 방식으로, 실제 차원 안전성을 갖고 동작한다:

```python
unit length(base: f64):
    mm = 0.001, cm = 0.01, m = 1.0, km = 1000.0

height = 175_cm
width  = 10_cm + 5_mm     # 공통 기준으로 자동 변환
# bad  = height + speed    # Error: Length + Velocity는 더할 수 없음
```

이것은 타입 시스템이 허용하면 LLM이 *결국에는* 저지를 바로 그 실수다. 그러니 타입 시스템이 허용하지 말아야 한다. Simple은 또한 맨 primitive로 가득 찬 public API를 — 프로덕션 문제가 되기 전에 — 품질 문제로 lint한다.

## 문서는 실행되어야 한다

생성 문서에는 특유의 실패 양상이 있다: **그럴듯하게 들린다.** 모델이 README 예제를 쓰고, API가 바뀌고, 아무도 예제를 갱신하지 않고, 6주 뒤 문서는 자신만만한 거짓말이 된다.

Simple의 **SDoctest**는 예제를 실행 가능하게 만든다. Markdown과 파일 로컬 예제가 테스트 러너를 통과하며, 출력과 일치해야 한다:

```python
>>> factorial(5)
120
>>> factorial(0)
1
```

```bash
simple test --sdoctest README.md
simple test --sdoctest src/math.spl
```

문서는 코드 *옆에 놓인* 산문이기를 멈추고, 유지보수되는 테스트 표면의 일부가 된다. LLM 생성물에서는 이게 큰 의미를 갖는다: 모델이 한 번도 실행되지 않은 사용법을 자신만만하게 지어낼 수 없다. 예제가 중요하다면, 실행 가능하게 만들어라.

## 아키텍처는 기억되는 게 아니라 검사되어야 한다

이것이 가장 깊은 실패다. 수천 줄 수준에서는 LLM이 국소적 이해를 흉내 낼 수 있다. 수백만 줄 수준에서는 시스템을 컨텍스트에 담을 수 없다. 하드웨어 레이어가 services를 호출하면 안 된다는 것, 어떤 모듈은 이미 검증됐으니 함부로 다시 쓰면 안 된다는 것을 더 이상 안정적으로 알지 못한다.

Simple은 아키텍처를 기계가 검사 가능하게 만든다. 아키텍처 인지 구조(MDSOC 캡슐과 매니페스트)를 지원하며, predicate 기반 pointcut으로 레이어 제약을 컴파일 타임 규칙으로 표현할 수 있다:

```python
forbid pc{ within(hal.*) }      "HAL cannot depend on services"
allow  pc{ within(services.*) } "Services can use HAL"
```

프롬프트는 *"아키텍처를 존중해 주세요"*라고 말한다. 컴파일러는 **안 됩니다**라고 말한다. 이것이 AI 생성 시스템에 맞는 종류의 제약이다 — 그리고 같은 pointcut 메커니즘이 일반 aspect(`before` / `after` / `around` advice)도 구동하며, 컴파일 타임에 weaving되고 aspect를 하나도 정의하지 않으면 오버헤드가 0이다.

## 원래 글이 놓친 부분: 모델이 잃어버릴 수 없는 기억

위의 네 가지 실패는 눈에 잘 띄는 것들이다. 하지만 조용히 가장 큰 피해를 주는 것은 *컨텍스트 손실*이고 — 이것을 고치려면 검사만으로는 부족하다. 모델의 작업 기억이 **사라지지 않는 곳**에 살아야 한다.

Simple의 두 가지 요소가 이를 정면으로 다룬다. 그리고 이 둘은 내가 처음에 과소평가했던 것들이다.

**런타임 패밀리(Runtime families).** 어떤 코드가 할당하고, 블로킹하고, GC를 돌릴 수 있는지를 모델이 기억해 주길 바라는 대신, Simple은 그 기대를 모듈 계약에 새겨 넣는다. 다섯 개의 패밀리가 있다:

| 패밀리 | 할당 | 변경 | 비동기 | 용도 |
| --- | --- | --- | --- | --- |
| `common` | heap (기본) | immutable | no | 순수 함수, 수학, 텍스트 |
| `nogc_sync_mut` | heap/arena/pool | mutable | no | 파일 I/O, 네트워킹, FFI, DB |
| `nogc_async_mut` | heap | mutable | yes | 비동기 I/O, 스레드, 액터 |
| `gc_async_mut` | GC 관리 | mutable | yes | GPU/CUDA, ML 파이프라인 |
| `nogc_async_mut_noalloc` | stack only | mutable | yes | 베어메탈, 임베디드, RTOS |

컴파일러가 *경계를 강제*한다: no-GC 컨텍스트에서 GC 모듈을 import하면 진단(diagnostic)이 나온다. 타깃 프리셋(`Baremetal`, `Hosted`, `EmbeddedWithHeap`)은 호환되는 패밀리로 모듈 해석을 자동 제한한다. 모델이 펌웨어에 실수로 가비지 컬렉터를 끌어들일 수 없다 — 모델이 신중해서가 아니라, 경계가 벽이기 때문이다.

**SDN 기반 프로젝트 데이터베이스.** 테스트, 할 일(todo), 버그, 대시보드가 모델의 프롬프트나 채팅 로그에 흩어지는 대신, 간결한 데이터 포맷을 통해 repo-native하게 저장된다. 이는 에이전트에게 *무엇이 존재하고 무엇이 검증됐는지*에 대한 실제로 질의 가능한 기억을 준다 — 시스템이 커질 때 LLM이 가장 먼저 잃어버리는 바로 그것이다.

이 둘을 합치면 "모델이 아키텍처를 잊어버렸다"가 반복되는 재앙에서, 시스템이 강제하는 제약이자 시스템이 보관하는 기억으로 바뀐다.

## repo 자체가 에이전트 하네스다

여기 내가 맨 앞에 내세웠어야 할 프레이밍이 있다: **Simple은 툴링에 의해 검사되기만 하는 게 아니다 — repo 자체가 AI 에이전트에 의해 운영되도록 만들어졌다.**

MCP 서버, LSP 서버, 에디터 플러그인을 탑재하고 있으며, 전부 Simple 자체로 작성됐다. 에이전트는 원시 텍스트에서 진단을 추측하지 않는다. *repo-native 툴체인에 직접 물어본다* — 빌드 결과, 테스트 실패, lint 경고, 디버그 정보, 코드 내비게이션을. repo는 에이전트 지침(`AGENTS.md`, `CLAUDE.md`)과 커넥터 설정(`.mcp.json`, 그리고 `.claude` / `.codex` / `.gemini` 진입점)을 in-tree로 가지고 있다.

그래서 개발 모델이 곧 thesis 전체를 구체화한다:

- **모델은 제안한다.**
- **툴체인은 검사한다** — 그리고 에이전트는 그 검사 결과를 분위기가 아니라 구조화된 도구를 통해 읽는다.
- **아키텍처는 명시적으로 남고**, 프로젝트의 기억은 repo 안에 산다.

## 별난 기능들은 무작위가 아니다

Simple은 표면적이 *매우* 넓다 — 매크로, 수학 블록(`m{}`, `loss{}`, `nograd{}`), autograd 경로, SFFI, VHDL 백엔드, GPU 커널, capability/effect 시스템, Lean 4 정형 검증. 이걸 다 나열하면 이야기는 *더 나빠진다*.

흥미로운 건 개수가 아니다. **그것들이 모두 같은 방향을 가리킨다는 점**이다: 의도를 실행 가능하게, 그리고 검사 가능하게 만들기. 파서 친화적 매크로는 컴파일러와 에디터가 *확장 전에* 추론할 수 있는 매크로다. 수학 블록은 실제 언어 구문으로 파싱되며, 검사된 식과 연결된 채로 텍스트·유니코드·LaTeX·Markdown으로 렌더링된다. 런타임 패밀리는 할당과 동시성을 계약의 일부로 만든다. 어느 것도 장식이 아니다 — 전부 모델에게 즉흥적으로 굴 여지를 줄이기 위한 것이다.

예를 들어, 검사된 수학 블록은 소스 가까이에 그대로 둘 수 있다:

```simple
loss = m{
    L = sum((y_pred - y_true)^2) / n
}
```

그리고 문서나 에디터 미리보기를 위해 LaTeX로 렌더링할 수 있다:

```latex
L = \frac{\sum (y_{pred} - y_{true})^2}{n}
```

## 마케팅 안개를 걷어낸 현재 상태

Simple은 실재하며, 정직하게 서술되어야 한다.

- **버전:** `VERSION` 파일은 `1.0.0-beta`로 표시되고, 최신 태그 릴리스는 **v1.0.0-beta**이다.
- **테스트 스냅샷 (2026-02-14):** 4,067 / 4,067 통과, 17.4초. 이는 스위트의 폭과 속도에 대한 증거이지 — 언어 대 언어 런타임 벤치마크가 *아니다*.
- **소스 규모 (2026-04-23):** Simple, Rust, C, 어셈블리 합쳐 약 227만 줄(주석 제외), 그중 약 178만 줄이 Simple 자체.

구현됐다고 광고해도 안전한 것들: Sspec, SDoctest, 커버리지, 추적성, 생성 스펙 문서, 시스템 테스트 mock 정책, self-hosted 컴파일러/인터프리터/로더, MDSOC 매니페스트, 파서 친화적 매크로, Tree-sitter 툴링, SDN 기반 데이터베이스, primitive-public-API linting, borrow-checking 인프라, watch/auto-build, 그리고 지원 ABI 서브셋에 대한 C/C++ 양방향 SFFI.

정확성을 위해 한 가지 경계를 분명히 해야 한다. Rust는 여전히 bootstrap seed와 호스트 구현 기반으로 존재한다. pure-Simple 컴파일러, 로더, 라이브러리, 생성 스펙 문서는 진행 방향이자 bootstrap이 검증하는 대상이지만, 오늘의 "pure Simple"을 "저장소에 Rust 소스가 없다"는 뜻으로 읽으면 안 된다.

한정어를 붙여 서술하는 게 나은 것들: **Lean 검증**은 지원 서브셋에 한해 완성됐고, **런타임 패밀리**는 지원 매트릭스로 한정되며, **LLVM 백엔드** 패밀리는 선언된 public 매트릭스 위에서 닫혀 있고, **VHDL 백엔드**는 문서화된 하드웨어 서브셋을 대상으로 하며(두 개의 GHDL RV32 시뮬레이션 레인 포함), **원격 베어메탈**은 8개의 권위 있는 레인을 갖지만 호스트·보드 인지적이고, **공유 UI 계약**은 완성된 범용 UI 레이어가 아니라 교차 표면 테스트 프로토콜이며, **수학 블록/autograd** 경로는 승격된 torch 기반 C/LLVM 범위에 한해 완성됐고 다른 백엔드는 보류 상태다.

생성 코드의 거짓말을 막기 위해 만든 언어가 과장으로 스스로를 마케팅해서는 안 된다.

## 직접 써 보기

```bash
git clone --recurse-submodules https://github.com/ormastes/simple.git
cd simple
export PATH="$PWD/bin:$PATH"
simple --version
```

기존 체크아웃에서는 예제/툴링 서브모듈을 초기화하거나 갱신한다.

```bash
git submodule sync --recursive
git submodule update --init --recursive
```

하드코딩된 바이너리 URL보다 소스 체크아웃을 가리키겠다. 현재 서브모듈 레이아웃을 함께 검증할 수 있고, 릴리스 패키징 과정을 강화하는 동안 오래된 플랫폼 asset 이름을 피할 수 있기 때문이다.

## 큰 기능 지도

| 영역 | 하는 일 | 상태 |
| --- | --- | --- |
| Self-hosted 툴체인 | 컴파일러, 인터프리터, 로더, CLI, 표준 라이브러리 툴링이 Simple 소스 레이어 안에 존재 | 구현됨. 광고해도 안전함 |
| 검증 워크플로 | Sspec, SDoctest, 커버리지, 추적성, 생성 스펙 문서, 품질 게이트가 테스트와 문서를 실행 가능한 증거로 만듦 | 구현되어 활성화됨 |
| 시스템 테스트 정직성 | 시스템 테스트 mock 정책과 anti-dummy 게이트가 가짜 초록 테스트와 placeholder 구현을 거부 | 주요 소스와 CLI 표면에서 활성 |
| 아키텍처 제어 | MDSOC 매니페스트, 가상 캡슐, 레이어 규칙, pointcut 제약으로 아키텍처를 기계가 검사 가능하게 만듦 | 구현됨. 계약 문서 있음 |
| 런타임 패밀리 | `common`, `nogc_sync_mut`, `nogc_async_mut`, `gc_async_mut`, `nogc_async_mut_noalloc`이 할당, 변경, 비동기, 타깃 가정을 표현 | 지원 매트릭스 범위에서 구현됨 |
| 파서 친화적 매크로 | 매크로 정의, 확장, 검증, hygiene가 raw text 치환이 아니라 컴파일러가 볼 수 있는 구조로 존재 | 구현됨 |
| 수학/autograd 블록 | `m{}`, `loss{}`, `nograd{}`가 언어 구문으로 파싱되고 text, Unicode, LaTeX, Markdown으로 렌더링됨 | 승격된 torch 기반 C/LLVM 범위는 완료. 다른 백엔드는 보류 |
| SDN 기반 데이터 | 프로젝트 메타데이터, 테스트, todo, 대시보드, DB형 기록이 repo-native 텍스트 데이터 흐름을 사용 | 구현됨 |
| AI/에디터 툴링 | MCP 서버, LSP 서버, Tree-sitter, 에디터 플러그인, 에이전트 지침이 repo를 직접 질의하게 함 | 지원 표면에서 구현됨 |
| SFFI | C/C++ 양방향 ABI 브리지, export/import, callback, layout 검사, proof test 제공 | 지원 ABI 서브셋에서 구현됨 |
| LLVM 백엔드 패밀리 | 공개 LLVM 매트릭스가 `stable` 또는 `unsupported` 행으로 닫혀 있음 | 한정어와 함께 구현됨 |
| Lean 검증 | 결정적 Lean 생성, proof inventory, 검사, 캐시 무효화, 검증 상태 보고 | 지원 서브셋에서 완료 |
| VHDL/베어메탈 레인 | VHDL-2008 백엔드, GHDL RV32 semihost/mailbox 시뮬레이션, QEMU, 원격 베어메탈 흐름 | 제한된 하드웨어 서브셋. 호스트/보드 인지적 |
| 공유 UI 계약 | Protocol V1 테스트 클라이언트와 공유 handler가 web backend와 TUI-web proxy 동작을 검증 | 범용 UI 레이어가 아니라 교차 표면 테스트 프로토콜 |
| GPU/CUDA/ML 경로 | GPU 커널, CUDA 지향 예제, ML/autograd 통합 | 유용하지만 백엔드별 지원 범위로 제한됨 |
