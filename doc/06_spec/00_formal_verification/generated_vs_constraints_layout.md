# Generated vs Constraints Lean Layout

Regeneration-safe layout for the Lean verification projects under
`src/verification/`. The GENERATED surface and HAND-WRITTEN proof work live in
separate files, so `simple gen-lean write` can never clobber manual theorems.

## Convention

For every generator-owned module `<Name>` (e.g. `MemoryCapabilities`):

| File | Owner | Rule |
|------|-------|------|
| `src/<Name>.lean` | `simple gen-lean` | 100% generator-owned. First lines carry the marker ``-- GENERATED by `simple gen-lean` -- DO NOT EDIT.``. Regeneration replaces it wholesale. |
| `src/<Name>Constraints.lean` | humans | `import <Name>`, re-opens `namespace <Name>`, holds ALL manual theorems/constraints. The generator never writes this file. |
| `*.byl` | BYL backend | Generated proof-model interchange. Regeneration may replace it, and it is not a checked proof by itself. |
| `lakefile.toml` | humans | Has one `[[lean_lib]]` per module plus one per `<Name>Constraints`, all listed in `defaultTargets`, so `lake build` checks both halves. |

Enforcement (fails loudly, never silently):

- `LeanCodegen.emit()` (`src/compiler_rust/lib/std/src/verification/lean/_Codegen/lean_codegen.spl`)
  prepends the marker header to every emitted file.
- Writers refuse to overwrite any existing file that lacks the marker — named
  error `GenLeanUnmarkedOverwrite` — even with `--force`:
  - `write_regenerated_files` in
    `src/compiler_rust/lib/std/src/verification/regenerate/__init__.spl`
  - `write_lean_files` in `src/compiler_rust/driver/src/cli/gen_lean.rs`
- If unsure whether a block is generated: treat it as manual and put it in
  `<Name>Constraints.lean`.

`src/verification/riscv_product` predates this and uses the equivalent
`src/RiscvProduct/Generated.lean` + `Constraints.lean` split governed by its own
`GENERATED_CONTRACT.md` (byl_backend pipeline, not `gen-lean`); it stays as-is.
The BYL backend is useful because it keeps compact product/profile/resource
facts cheap to regenerate, but any SPipe or release evidence that cites BYL must
also cite the durable Lean consumer (`Constraints.lean` or the lane wrapper) and
the exact check command that passed after regeneration.

## Regeneration and build commands

```bash
bin/simple gen-lean write [--project <name>] [--force]   # regenerate (Generated files only)
bin/simple gen-lean compare                              # diff generated vs disk
sh scripts/check/check-lean-proofs.shs                   # lake build every project + sorry/axiom scan
cd src/verification/<project> && lake build              # one project
```
