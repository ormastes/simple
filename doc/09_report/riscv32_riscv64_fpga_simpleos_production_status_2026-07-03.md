## Change Landed In This Slice

The RTL smoke wrappers now resolve the repository root correctly and the top
level smoke script reports both RV32 and RV64 lane failures in one run instead
of stopping after the first missing artifact.

The generated Linux GHDL runners now use per-run generated bundle roots
(`$GEN_DIR/rv32/rtl` and `$GEN_DIR/rv64/rtl`) instead of directly compiling the
handwritten example RTL tree. The bundle generator has a small executable
entrypoint and no longer depends on the deleted handwritten RTL tree for its
bundle output.

Minimal RV32 and RV64 smoke assembly payloads are present under
`examples/09_embedded/fpga_riscv/sw/`, and the RV64 public smoke wrapper no
longer requires an unrelated native-build-capable compiler before running the
GHDL lane.

`scripts/check/check-riscv-fpga-simpleos-preflight.shs` now checks the existing
RV64 preflight plus RV32 FPGA artifacts, VHDL template generation,
executable-core evidence, load/run evidence, and decoded-marker GHDL evidence,
so production status cannot ignore the 32-bit lane.

RV32I RTL helper visibility is fixed for the existing `rv32i_rtl` core path:
`src/lib/hardware/rv32i_rtl/lsu.spl` exports the LSU helpers and
`core.spl` uses the same brace import form as the rest of the lane. Direct VHDL
compile now reaches the real remaining blocker: the behavioral core has no
`@hardware` boundary for `compile --backend=vhdl`.

`scripts/fpga/generate_rv32_vhdl.shs` still writes the generated RV32
decode/control helper, and now also copies a minimal RV32/C executor template
into `build/vhdl/rv32/rv32_exec_core.vhd` with a simulation top at
`soc_top_rv32_sim.vhd` and K26 wrapper at `soc_top_rv32.vhd`. The combined

old decode helper. `sh scripts/fpga/ghdl_validate_rv32.shs --simulate` now
preloads `build/os/simpleos_riscv32_fpga.elf` through a flat binary image and
requires the decoded `FPGA-RV32` marker. The executor now separates the

path (`43733..43739`), avoiding the previous 24K-word distributed RAM mapping.
Full Vivado RV32 implementation now passes DRC/routing/bitgen and writes the
fresh KV260 bitstream consumed by preflight.

`simple os build --scenario=riscv64-fpga-mmode` and

The K26 Vivado wrapper now uses the release Simple binary fallback, regenerates
the RV64 VHDL/TCL every run, emits a K26 UART XDC constraint, preserves Vivado
failure status despite tailing logs, and refuses to run Vivado on placeholder
core RTL unless `ALLOW_PLACEHOLDER_RTL=1` is set for plumbing diagnostics. It

STARTUPE3 wrapper that supplies internal clock/reset so only `uart_tx` is a

keeps an XSDB failure log at `build/fpga/k26/load_elf_k26.log`.

The RV32/RV64 SimpleOS preflights now fail stale bitstream evidence explicitly:
the programmed bitstream must be newer than the generated preload/core/top RTL
inputs before it can count as production FPGA evidence. RV64 now passes that
freshness gate after a full Vivado rebuild. RV32 now also passes that freshness
gate after the executor scratch window was reduced and the full Vivado build

supports `ILA_TRIGGER_PROBE`, `ILA_TRIGGER_BYTE`, and `ILA_TRIGGER_POSITION`,
and the K26 debug build wrappers accept `ILA_DATA_DEPTH`. An earlier deep RV32 debug

`70,80,71,65,45,82,86,51,50` (`FPGA-RV32`) as the first bytes.

The RV32 generator now stamps an absolute `rv32_payload.mem` path into the
generated executor VHDL. This fixes the Vivado synthesis failure exposed by a

The first post-fix KV260 ILA runs did not prove RV32: marker-latch capture

after the first byte and enters later byte calls correctly, but `uart_put_byte`

current RTL mirrors all byte lanes in the tiny scratch window, and the fresh
KV260 ILA capture now proves the full `FPGA-RV32` marker.
