## Accepted evidence (grounded directly in each commit's own message + source at HEAD)

- **SHOWCASE_RESOLUTION/SHOWCASE_DPI convention landed across all four render entry points** (`297815ba0a`, `2076527bb1`, `9b42b2011d`): `SHOWCASE_RESOLUTION` unset/`"4k"` => 3840x2160 (new DEFAULT, was a small size before), `"8k"` => 7680x4320, or explicit `"WxH"`; `SHOWCASE_DPI` default 300, applied for graphics/widget (stroke/detail/text scale), read-but-not-applied for web (engine has no DPI hook — `doc/08_tracking/bug/web_render_dpi_scaling_unsupported_2026-07-12.md`, filed open, feature gap not regression).
- **Cross-module name-leak blank-render bug fixed** in `graphics_2d_showcase_gui.spl` (`2076527bb1`): a `use graphics_2d_showcase.{run_graphics_2d_showcase, draw_showcase}` import leaked `draw_showcase`/`label` into the importing module's scope, shadowing local same-named mutating functions and silently discarding their effect on the caller's `Engine2D` instance. Renamed locals to `wm_draw_showcase`/`wm_label`; verified 1,439,967/1,440,000 nonzero (800x600). Underlying compiler root cause (why an explicit `{A}` import list leaks other same-named top-level symbols, and why the leaked symbol wins a name-resolution race against a local definition) remains open: `doc/08_tracking/bug/cross_module_imported_fn_mutation_not_propagating_2026-07-12.md`.
- **SimpleOS WM boots a real 3840x2160 argb8888 screen** (`603fabe601`): `gui_entry_desktop.spl` framebuffer 1024x768 -> 3840x2160, QEMU `-global VGA.vgamem_mb=64`. Root cause fixed: `rt_alloc`/`rt_alloc_zeroed` silently truncated allocations over 16MB to 16MB while reporting success, overflowing the 33.2MB 4K backbuffer into adjacent heap; both clamps removed. **Commit message claims** harness `status=pass`: scanout `3840x2160` stride `15360`, `baseline_ppm=24883217 bytes`, `changed_bytes=23174856`, `baseline_sha256==restored_sha256`. See Discrepancy section below — this could not be independently reproduced in this verification pass.
- **8K SimpleOS screen explicitly out of scope**, filed as follow-up: 132MB VRAM requirement exceeds BGA/std-vga capability.
- **A freestanding-codegen bug was hit and worked around** while making the 4K resolution a named constant in `gui_entry_desktop.spl`: hoisting a `val: u32` to module level and reading it from `spl_start()` corrupts the value under the `x86_64-unknown-none` Cranelift target (`width=16000 height=1048` garbage instead of `3840x2160`); plain local literals do not reproduce it. Filed, workaround (local literals) is what ships: `doc/08_tracking/bug/x64_freestanding_module_level_val_u32_desktop_gui_2026-07-12.md`.
- **WebIR/DrawIR optimization plan committed** (`3963a47`, `doc/03_plan/ui/webir_drawir_optimization.md`, 236 lines, draft/not executed): identifies `simple_web_layout_render_html_draw_ir` (Path B) and `draw_ir_diff_compositions` (node-level diff, spec-tested, zero production callers) as existing building blocks; proposes wiring the diff into the content-frame render loop so repaint cost scales with changed nodes, not total pixels — the concrete response to the ~55-60x full-engine-vs-tag-strip perf gap filed in `web_render_full_engine_content_frame_reroute_perf_2026-07-12.md`.
- Canonical catalog (`src/lib/common/ui/showcase_catalog.spl`) still contains exactly three stable identities with an explicit 3x3 readiness matrix, and every readiness bit is correctly `false`.

## Discrepancies found during verification (flagged, not papered over)

- **Web 4K uniform-frame bug: doc says "Fixed", but the fix is not in the pushed source.** `doc/08_tracking/bug/web_render_budget_env_override_ignored_by_stage_rearm_2026-07-12.md` (committed in `297815ba0a`, part of the pushed range) is headed "Status: Fixed" and cites a fix in `simple_web_html_layout_renderer.spl`. Verified directly: `git show HEAD:src/lib/gc_async_mut/gpu/browser_engine/simple_web_html_layout_renderer.spl` still has the un-overridden, pre-fix `resolved_budget_ms = if budget_ms <= 0: WEB_RENDER_BUDGET_MS else: budget_ms` logic — no `env_override`, no area-scaled floor. The actual fix exists only as an uncommitted `M` change to that file in this working tree. **The commit message for `297815ba0a` says this explicitly**: *"the 8-line engine fix commits separately (WC currently contaminated by a peer layout refactor)"* — i.e. the fix is real, reviewed-in-intent, and blocked from landing cleanly by unrelated concurrent working-tree changes, not abandoned. Net effect for anyone reading only the bug doc: the "Fixed" header overclaims relative to what's actually reachable at this repo's HEAD/origin. Treat the web-4K-degrades-to-uniform-frame behavior as **still live** until a clean follow-up commit lands.
- **SimpleOS 4K-screen PASS could not be independently reproduced.** Neither the committed evidence report nor a live rerun in this pass shows the PASS state the `603fabe601` commit message describes. The committed `doc/09_report/simpleos_wm_fullscreen_evidence_2026-07-12.md` shows `status: fail, reason: dynamic-scanout-or-desktop-readiness-missing`, scanout correctly `3840x2160`/stride `15360`, but `baseline_ppm_bytes=0` / all PPMs 0 bytes. A live rerun of `scripts/check/check-simpleos-wm-fullscreen-evidence.shs` during this verification pass (via the live `build/simpleos_wm_fullscreen_evidence/evidence.env` artifact) reproduced the **same failure signature** — correct 4K scanout, zero-byte PPMs, `status=fail`. This is the same "stale/inconsistent evidence file" pattern already flagged in the 2026-07-11 verify report for a different run; here it additionally means the specific PASS numbers in the commit message (`changed_bytes=23174856`, sha256 round-trip) are **reported-in-the-commit-message only** as of this pass, not confirmed against any artifact this pass could inspect or reproduce. The rt_alloc/rt_alloc_zeroed fix and the 3840x2160 scanout change are real (confirmed by reading the diff), but the specific interactive-PASS claim needs a fresh run + a regenerated report before it can be cited as current evidence.
- **Host-WM default launch behavior now silently attempts full 4K.** None of `wm_graphics_2d_showcase_client_env`/`wm_web_standards_showcase_client_env` (`src/lib/common/ui/wm_app_process_contract.spl`) set `SHOWCASE_RESOLUTION` for the spawned child, and `widget_showcase_gui.spl`'s `run_wm_client()` calls `build_wm_pixels(W, H, ...)` with the hardcoded `W=3840, H=2160` directly, bypassing its own resolver entirely. So launching any of the three host-WM wrapper scripts today, with no env override, attempts a full 3840x2160 child render under the interpreter (the only reachable lane — `bin/simple` is still exit-132-blocked) — the same "compiled-lane-gated"/"~5k px/s" wall documented for standalone. This is a real, reachable regression path introduced by today's default-resolution change, not previously true when the small size was the default.
- **"Small-verified" claims for host-WM graphics/web could not be confirmed at the adapter level.** The 1,439,967/1,440,000 nonzero verification for the graphics cross-module fix is real but was performed at 800x600 (explicit override), and no report in this repo shows the `wm_graphics_2d_showcase_gui.spl`/`wm_web_standards_showcase_gui.spl` wrapper scripts actually launched end-to-end (winit window + compositor blit) at any resolution. Do not read "bug fixed" as "adapter GUI-verified."
