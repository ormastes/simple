# rv32 FPGA — Linux-loading integrity + sustained hard-job soak (2026-07-22)

Lane OO. Two proofs on the **FPGA flow**, run as GHDL simulation of the PROVEN
`rv32_exec_core.vhd` — the FPGA-exact RTL that closes timing and reaches
`DONE=HIGH` on the Kria K26 (see MEMORY: rv32 boot→login→ls PROVEN in GHDL,
K26 timing MET). RTL was treated as READ-ONLY and pinned to this worktree's
origin copy (`examples/09_embedded/fpga_riscv/rtl/rv32_exec_core.vhd`); Lane KK
was editing that file in parallel and none of its edits were used here.

Both gates are additive (NOT wired into CI / the landed 9/9 hardware gates).

## What the FPGA load path actually is
The rv32 core's memory is on-die BRAM initialized from `.mem` files:
- code ROM `0x80000000..0x8000FFFF` (64 KB) from `rv32_payload.mem`
- data ROM `0x40000000..0x4000FFFF` (64 KB) from `rv32_fat32.mem`
- UART TX: store-byte to `0x10000000`

In simulation the `.mem` preload **is** the on-hardware mechanism: the bitstream
bakes the identical words into the same BRAM via `INIT_xx` strings. So the
integrity proven in sim is exactly the integrity of the hardware load path.

## Gate 1 — Linux-loading integrity (`scripts/fpga/check_linux_loading_rv32.shs`)
A deterministic 64 KB payload (Park-Miller LCG, exact in both awk and the core)
is preloaded into the data ROM. The rv32 core sums all 16384 words with 32-bit
wrap and emits the result over UART as `CS=XXXXXXXX~`. The host computes the
identical 32-bit sum. A match proves every byte of the payload was carried
through the real load path intact (any dropped/corrupted word changes the sum).

This is a **LOADING** gate, not a boot: rv32 has no MMU and cannot execute a
Linux Image — the honest claim is full-capacity payload delivery with verified
end-to-end integrity.

| metric            | value                         |
|-------------------|-------------------------------|
| payload           | 65536 bytes (16384 words, full data ROM) |
| core checksum     | `883FEE27`                    |
| host golden       | `883FEE27` (match)            |
| core cycles       | 341,416                       |
| wall time         | 13 s                          |
| effective sim rate| 26,262 Hz                     |

Result: `CHECK_LINUX_LOADING_RV32: PASS`.

## Gate 2 — sustained hard-job soak (`scripts/fpga/soak_rv32_hard_job.shs`)
An Adler-32 reduction over the byte sequence `(i & 0xFF)` for `i` in
`[0, COUNT)`. Each iteration issues two `remu` operations, driving the core's
multi-cycle divider (the unit Lane KK fixed) continuously. Progress markers
(`P`) are emitted every 10000 iterations; the final Adler-32 golden is emitted
as `DONE=XXXXXXXX~`. Gate asserts: wall ≥ 60 s sustained, golden == host
reference, every UART byte in the expected charset (zero garbage), progress
markers present (no early livelock). Correct golden = the full COUNT iterations
ran, which itself rules out stall/livelock.

Default `COUNT=90000` (env `RV32_SOAK_COUNT`). On this dev box the run is 171 s,
a **2.85× margin** over the 60 s floor: the gate passes as long as the host's
GHDL sim rate is under ~2.8× this box's, and stays under the 10-min ceiling
until ~3.5× slower — so 171 s is well-centered. A materially faster box that
drops the wall below 60 s should raise `RV32_SOAK_COUNT` (that env knob exists
for exactly this); a sub-60 s wall on a fast box is a re-tune, not a broken gate.

| metric            | value                         |
|-------------------|-------------------------------|
| iterations        | 90,000                        |
| golden (COUNT=90000) | `A77902FA` (core == host)  |
| progress markers  | 9                             |
| core cycles       | 7,931,241                     |
| wall time         | **171 s** (2.85 min, ≥ 60 s floor) |
| effective sim rate| 46,381 Hz                     |
| iteration cost    | ~122 core cycles/iter (2× `remu`) |

Result: `SOAK_RV32_HARD_JOB: PASS`.

## Blockers (explicit — not silently scoped away)

### Load-path capacity: 64 KB, not 22 MB (BRAM ceiling, NOT wall-time)
The user directive mentioned an 8–22 MB Linux Image. This rv32 SoC has **no
external DRAM and no memory controller/AXI** — code and data live entirely in
on-die BRAM (64 KB each). 22 MB cannot be loaded because the storage does not
exist on this core, not because simulation would be slow. The largest payload
the real load path physically carries is the full 64 KB data ROM, which Gate 1
exercises end-to-end. Loading a multi-MB image is a board bring-up item gated on
adding a DDR controller + AXI interconnect to the SoC (out of scope here, filed
as the honest ceiling).

### Board soak blocker: PMOD UART adapter (H12 / E10)
The soak and loading gates run in GHDL because the on-board serial capture path
is blocked: the Kria K26 / KV260 carrier routes the rv32 core UART TX to PMOD
pins **H12** and **E10**, and the physical PMOD-to-UART adapter needed to sample
those pins on the bench is not currently wired. Until that adapter is in place,
a board-side soak (bitstream program → real-silicon Adler-32 → serial golden
capture) cannot be captured to the board-evidence bar. This is a hardware-wiring
gap, not an RTL or toolchain gap — the same `rv32_payload.mem` bitstream BRAM
init drives both sim and board, so the sim result is the board-runnable proxy
pending the adapter.

## Reproduce
```sh
sh scripts/fpga/check_linux_loading_rv32.shs      # ~15 s,  Gate 1
sh scripts/fpga/soak_rv32_hard_job.shs            # ~3 min, Gate 2
```
Deterministic: fixed seeds/sizes; each script ends `<NAME>: PASS` or a precise
`FAIL:`. Toolchain: `riscv64-unknown-elf-gcc` (rv32im multilib) + GHDL 4.1.0.
