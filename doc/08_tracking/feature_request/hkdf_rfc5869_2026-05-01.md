# Feature Request: HKDF RFC 5869 Implementation

**Date:** 2026-05-01
**Requested by:** hkdf_rfc5869_spec agent

## Summary

Add HKDF (HMAC-based Key Derivation Function) as defined in RFC 5869 to
`src/lib/common/crypto/`.

## Motivation

- Required for TLS 1.3 key schedule, JWT key derivation, and other modern
  cryptographic protocols.
- The tree already has `hmac.spl`, `sha256.spl`, and `sha512.spl` — HKDF is a
  thin two-step wrapper over HMAC and is a natural addition.
- RFC 5869 Appendix A provides canonical test vectors for verification.

## Proposed API

```simple
# src/lib/common/crypto/hkdf.spl

# Step 1: Extract — PRK = HMAC-Hash(salt, IKM)
# If salt is empty, use HashLen zero bytes as salt per RFC 5869 §2.2
fn hkdf_extract(salt: [u8], ikm: [u8]) -> [u8]

# Step 2: Expand — OKM = T(1) || T(2) || ... truncated to `len` bytes
# len must be <= 255 * HashLen (8160 for SHA-256)
fn hkdf_expand(prk: [u8], info: [u8], len: i64) -> [u8]

# Combined convenience wrapper using SHA-256
fn hkdf_sha256(salt: [u8], ikm: [u8], info: [u8], len: i64) -> [u8]
```

## Spec File (blocked on impl)

`test/unit/lib/crypto/hkdf_rfc5869_spec.spl` — ready to add once impl lands.
Six core specs: A.1 PRK, A.1 OKM, A.2 PRK, A.2 OKM, A.3 PRK, A.3 OKM.
Optional: L=1, L=8160 (255×32 max) boundary tests.

## Notes

- HKDF-SHA-512 variant may follow naturally from the same structure.
- No new `rt_*` externs needed — builds entirely on existing `hmac.spl`.
- The `pbkdf2.spl` already in tree is a related KDF and can serve as a
  structural reference.
