## Facts

- `tb_rv64_wb_soc_smoke.vhd` instantiates `entity work.soc_top_rv64`.
- No tracked `.vhd` file anywhere in the repo defines `soc_top_rv64`
  (repo-wide grep at `61fb4f0d884`).
- The similarly named `soc_top_rv64_k26` (e.g.
  `scripts/fpga/build_k26_vexriscv.shs:56`) is a **different** entity: the
  Vivado top of the VexRiscv board flow — an external core, not a
  Simple-generated artifact, and not this entity.
- The backend-generated lane that could plausibly produce it
  (`build/os/rtl/`) does not exist in a clean checkout, and `rv32i_rtl/`
  / `rv64gc_rtl/` contain zero checked-in `.vhd` (VHDL there is
  backend-generated by design).

So a clean checkout cannot elaborate this testbench from tracked files alone —
exactly the reproducibility-failure class the research doc names
(`doc/01_research/hardware/riscv/riscv32_riscv64_unification_realrtl_aop_jtag_2026-07-21.md`
§3.3, §6.2: "wrapper around a missing entity is rejected as CPU evidence").

## Required fix (do NOT weaken checker rule 4)

Either:

1. Make the artifact set complete: generate/track `soc_top_rv64` in the same
   artifact set the testbench ships in, with provenance; or
2. Move/gate the testbench until the generated lane that provides the entity
   exists, keeping it visibly classified as `fixture` awaiting its DUT.

The checker staying RED on the current tree is correct fail-closed behavior;
this doc is the tracked justification, not an excuse to exclude the file.
