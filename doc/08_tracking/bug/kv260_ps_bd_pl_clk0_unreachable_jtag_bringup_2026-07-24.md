# KV260 PS block-design `pl_clk0`/`pl_resetn0` unreachable in JTAG-only bring-up

- **Date:** 2026-07-24
- **Severity:** high (blocks all PS-clocked fabric designs on KV260 for
  JTAG-only bring-up; rv64 bring-up was built on this assumption)
- **Status:** root-caused; resolution is architectural (switch clocking
  source), not a register-poke fix

## Symptom

An rv64 SoC design on KV260 clocked its fabric core from the Zynq
UltraScale+ PS block design's `pl_clk0` (with `pl_resetn0` as the fabric
reset), following the PS block-design pattern used for `pl_clk0` export in
`build_k26_rv64.shs`. In a **JTAG-only bring-up** (attaching over JTAG
without first booting the PS through FSBL/U-Boot/Linux), the fabric core
never runs: no UART output, no BSCAN tap activity beyond static/idle reads,
and the design otherwise behaves as if held in permanent reset.

## PS register evidence gathered (xsdb pokes, none fixed it)

Each of the following was tried, over xsdb, in an attempt to force
`pl_clk0` active without a full PS boot. All were applied successfully
(register writes took) but **none caused `pl_clk0` to toggle or
`pl_resetn0` to de-assert**:

| Poke | Value | Intent | Result |
|------|-------|--------|--------|
| `PL0_REF_CTRL` | `0x01010A00` | Select 100 MHz PL0 reference clock, sourced from IOPLL | No effect — IOPLL itself never reaches lock in this flow (it is configured/locked by the BootROM during normal PS boot, which never runs here) |
| `RST_LPD_IOU2` bit 18 clear | — | Un-reset the GPIO block (prerequisite for further EMIO pokes) | GPIO un-reset succeeded; no effect on PL clock/reset path |
| `DIRM_5`/`OEN_5`/`DATA_5` bit 31 | `0xFF0A0344`/`48`/`54` | Drive EMIO95 high (attempt to fake a "PS ready" style signal) | EMIO95 driven high; no effect on `pl_clk0`/`pl_resetn0` |
| `psu_post_config` | — | Standard post-configuration hook run after PL bitstream load | Ran; no effect — `pl_clk0` still not toggling afterward |

**Full `psu_init` (the complete PS bring-up sequence, as normally run by
FSBL/PMU firmware) HANGS in `psu_pll_init_data` when driven by hand over
JTAG.** This confirms PS clock-generator bring-up is not something that can
be shortcut with a handful of register pokes in a JTAG-only harness — it
requires the real boot sequence to run to completion.

## ILA proof (netlist-level, definitive)

A netlist ILA was inserted post-route (see
`scripts/fpga/insert_ila_netlist.tcl`, and the "Fabric-internal visibility"
section of `doc/07_guide/hardware/fpga/simple_riscv_jtag_debugging.md`) with
its own clock domain independent of the PS, probing the fabric core clock
net (`clk_core`). Recipe: open the `impl_1` **ROUTED** checkpoint (the
`synth_1` DCP fails DRC INBB-3 — instantiated black box — because the PS
block design is synthesized out-of-context and only resolves post-route),
`create_debug_core u_ila_0 ila`, connect clock + `MARK_DEBUG`-tagged probe
nets, `implement_debug_core` → `place_design` → `route_design` →
`write_bitstream` + `write_debug_probes`.

**Result: `clk_core` never toggles.** This proves — rather than merely
suggests — that `pl_clk0` does not reach fabric in a JTAG-only bring-up on
this board: the PS clock/reset generation logic that drives `pl_clk0` is
itself gated by PS boot progress the JTAG-only flow never produces.

## Root cause

The Zynq UltraScale+ PS's PL clock outputs (`pl_clk0`..`pl_clk3`) and their
associated resets (`pl_resetn0`..`pl_resetn3`) are generated by PS-internal
clock/reset controllers that are themselves gated behind PS boot state (IOPLL
lock, FPD/LPD power-domain and reset-domain sequencing normally driven by the
BootROM → FSBL → PMU firmware chain). On KV260, with the A53 cores held in
POR (no boot image, no PMU firmware, JTAG-only attach), that sequencing never
runs, so `pl_clk0` stays gated off and `pl_resetn0` stays asserted (low)
indefinitely. No combination of individual xsdb register pokes reproduces
what the full boot sequence does, and driving the full sequence (`psu_init`)
by hand over JTAG does not complete either (hangs in
`psu_pll_init_data`).

## Resolution

**PS-BD clocking is unusable for JTAG-only bring-up on this board.
Free-running `CFGMCLK` is the required clocking source** — this is the same
pattern already proven and in production for the rv32 build
(`generate_rv32_vhdl.shs` / `build_k26_rv32.shs`). Any KV260 design intended
for JTAG-only bring-up (i.e. not booting the PS) should be clocked from
CFGMCLK from the start rather than from the PS block design's `pl_clk0`.

The project-flow ILA knob that existed alongside the PS-clocked rv64 build
(`ENABLE_UART_ILA`/`ila_reset_release_count`/`ila_data_depth` in
`scripts/fpga/build_k26_rv64.shs`) was found to be dead code (set, never
consumed — no `create_debug_core` call existed) while investigating this;
it has been deleted. The netlist-level ILA insertion recipe above
(`scripts/fpga/insert_ila_netlist.tcl`) replaces it as the supported way to
get fabric-internal visibility on this board.

## Improvement note (not implemented — filed only)

The `uart_bscan_log` tap used for board debug is **read-only** (TDI is
ignored by the design). All-zero reads over this tap are ambiguous: they
cannot distinguish a dead/unclocked tap from a live tap that simply has no
data to report. A future RTL revision should add an always-running,
clock-domain aliveness counter into the capture word so a live tap reads as
unambiguously alive even with zero payload data. This was **not** implemented
as part of this investigation because a build was in flight at the time
(RTL churn during an active build risks corrupting that build); it is
recorded here for a follow-up change.
