# Bug: Ed25519 implementation does not match RFC 8032 §7.1 byte-for-byte

- **Date:** 2026-05-01
- **Status:** Open
- **Module:** `src/os/crypto/ed25519.spl`
- **Found by:** RFC 8032 §7.1 test vector spec (FF agent run, draft spec at `test/unit/lib/crypto/ed25519_rfc8032_spec.spl` left uncommitted)

## Symptom

Running RFC 8032 §7.1 test vectors against the existing Ed25519 implementation:

```
15 examples, 8 failures

PASS  T1: signature verifies under the correct public key
PASS  T1: signature rejected under different public key
PASS  T1: signature rejected when first byte flipped
FAIL  T1: derived public key matches RFC 8032 §7.1 vector
FAIL  T1: sign(empty) byte-matches RFC 8032 §7.1 expected signature

(same pattern for T2, T3, T_SHA_ABC: verify-under-correct-pubkey passes,
 derived-pubkey-matches-RFC and sign-byte-matches-RFC fail)
```

## What this means

- **The implementation is internally consistent.** sign-then-verify with the same key works. A signature signed with `k` verifies under the public key derived from `k`.
- **The implementation is NOT RFC 8032 compatible.** Public keys derived from the same seeds do not match RFC 8032 §7.1 reference values, and signatures over the same messages do not byte-match RFC 8032 §7.1 reference signatures.

This is the worst kind of crypto bug: it self-tests as working, but cannot interoperate with any RFC-compliant Ed25519 implementation (OpenSSH `ssh-ed25519`, OpenSSL EVP_PKEY_ED25519, libsodium `crypto_sign_detached`, JWT EdDSA, etc.).

## Likely cause (hypotheses to investigate)

Per RFC 8032 §5.1.5 the public key derivation from a 32-byte seed is:

```
1. h = SHA-512(seed)        # 64-byte digest
2. a = first 32 bytes of h
3. a[0]  &= 248             # clear bottom 3 bits
4. a[31] &= 127             # clear top bit
5. a[31] |= 64              # set second-from-top bit
6. A = a * B  on Curve25519 # scalar multiply by base point
7. encode A as 32 bytes
```

The most common bugs that produce the observed "internally consistent but not RFC" pattern:

1. **Treating the seed as a raw scalar** — skipping the SHA-512 step. Self-consistent (any consistent transform works for sign/verify) but wrong.
2. **Forgetting the bit-clamping** at steps 3-5. Self-consistent, but produces public keys off the canonical subgroup.
3. **Wrong endian-ness of A's encoding** at step 7. Self-consistent if both sign and verify use the same wrong endian.
4. **Hash signing path uses SHA-256 instead of SHA-512.**

## How to confirm

Read `src/os/crypto/ed25519.spl::ed25519_public_key` (or whatever the derivation function is named). The RFC algorithm has all 7 steps above; if any are missing or wrong, you've found the bug.

## Reference values that should hold

For the canonical seed `9d61b19deffd5a60ba844af492ec2cc44449c5697b326919703bac031cae7f60`:

- RFC 8032 §7.1 expected public key: `d75a980182b10ab7d54bfed3c964073a0ee172f3daa62325af021a68f707511a`
- RFC 8032 §7.1 expected sig over empty message: `e5564300c360ac729086e2cc806e828a84877f1eb8e5d974d873e065224901555fb8821590a33bacc61e39701cf9b46bd25bf5f0595bbe24655141438e7a100b`

If `ed25519_public_key([0x9d, 0x61, 0xb1, ...])` returns anything other than `d75a98...`, that's the bug.

## Impact

- **Cannot interoperate with any RFC-compliant Ed25519 stack.**
- `ssh-ed25519` host keys, user keys, and signatures generated by this impl will be rejected by OpenSSH.
- JWT `EdDSA` signed by this impl will be rejected by `jose`/`jsonwebtoken`/`pyjwt`.
- TLS 1.3 / certificate signatures using Ed25519 will fail.

In practice this means the existing Ed25519 module is a **toy implementation** that cannot be used for real interop. Any caller that requires standards conformance (SSH, JWT, TLS, certificate signing) is currently broken.

## Resolution path

1. Read `src/os/crypto/ed25519.spl` end-to-end.
2. Compare the public-key derivation against RFC 8032 §5.1.5 (the 7 steps above).
3. Find the deviation. Fix.
4. Commit the spec at `test/unit/lib/crypto/ed25519_rfc8032_spec.spl` once vectors pass.
5. Audit all callers of `ed25519_*` (likely SSH host-key code in `src/os/apps/sshd/`) for the same kind of self-consistent-but-not-RFC trap.

## Why the spec was not committed

Per project rules ("NEVER skip failing tests"), committing a spec with 8 known-failing assertions would either (a) leave the suite red, or (b) require `skip()` on the failing tests, which the project rules forbid. The right move is to file this bug now, fix the impl, then commit the spec once green.
