# Ed25519 Verify Rejects SSH Exchange-Hash Signatures

**Filed:** 2026-06-06
**Status:** Open
**Area:** `src/os/crypto/ed25519.spl`, SSH host-key tests

## Summary

`ed25519_sign(seed, pubkey, message)` produces 64-byte deterministic signatures for SSH 32-byte exchange hashes, and the live SSH/OpenSSH login path accepts the server signature. Local verifier checks with `ed25519_verify(pubkey, message, sig)` reject at least the synthetic `_test_exchange_hash()` signature generated by `ssh_sign_exchange_hash`.

`rt_ed25519_verify` also rejects the generated SSH signature in both observed argument orders:

- `rt_ed25519_verify(hash, pubkey, sig)`
- `rt_ed25519_verify(pubkey, hash, sig)`

## Reproduction

Use `test/01_unit/os/apps/sshd/ssh_kex_hostkey_matrix_spec.spl` helpers to sign `_test_exchange_hash()` with `ssh_sign_exchange_hash("ssh-ed25519", ...)`, extract the raw signature, and verify it with `ed25519_verify(pubkey, hash, raw_sig)`.

Expected: verifier accepts the signature.

Actual: verifier returns `false`.

Attempted regression assertion on 2026-06-06:

```simple
val hash = _test_exchange_hash()
val raw_sig = _decode_sig_blob(ssh_sign_exchange_hash("ssh-ed25519", host_keys, hash).unwrap()).1
expect(ed25519_verify(pubkey, hash, raw_sig)).to_equal(true)
```

Result:

- `bin/simple test test/01_unit/os/apps/sshd/ssh_kex_hostkey_matrix_spec.spl --clean --json`
- `total_passed: 28`, `total_failed: 1`

## Related Evidence

The broader pure-Simple Ed25519 primitive is still not RFC8032-clean on this worktree:

- `bin/simple test test/01_unit/lib/crypto/ed25519_rfc8032_spec.spl --clean --json`
- `total_passed: 4`, `total_failed: 11`

That suite shows canonical short-message verifies can pass, while derived public-key/signature vector checks and some rejection checks still fail. This SSH bug should remain open until the positive exchange-hash verifier assertion can be restored without breaking the normal SSH matrix.

## Current Mitigation

The SSH host-key matrix validates SSH blob structure, deterministic signing, nonzero signatures, transcript sensitivity, and tamper rejection where the local verifier path is reliable. It does not require local verifier acceptance for every generated SSH exchange-hash signature until this verifier gap is fixed.
