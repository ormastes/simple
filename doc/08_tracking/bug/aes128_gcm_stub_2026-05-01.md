# Bug: aes128_gcm.spl declares externs not registered in interpreter or runtime

**Date:** 2026-05-01
**Status:** Open
**Component:** `src/os/crypto/aes128_gcm.spl`

## Summary

`src/os/crypto/aes128_gcm.spl` is source-complete (full pure-Simple AES-128-GCM
implementation) but is unrunnable in interpreter mode because four of its `extern fn`
declarations have no backing implementation in either the interpreter dispatch table or
the compiled runtime.

## Missing Externs

Declared in `src/os/crypto/aes128_gcm.spl` lines 13–16:

```
extern fn rt_aes_sbox(i: i64) -> i64
extern fn rt_aes_rcon(i: i64) -> i64
extern fn rt_aes128_encrypt_block_into(key: [u8], block: [u8], out: [u8]) -> i64
extern fn rt_tls13_aes128_gcm_encrypt(key: [u8], nonce: [u8], plaintext: [u8], aad: [u8]) -> [u8]
```

None of these appear in:

- `src/compiler_rust/compiler/src/interpreter_extern/mod.rs` — only
  `rt_aes_encrypt_block_with_expanded` and `rt_aes_decrypt_block_with_expanded`
  are registered (lines 813–814).
- `src/compiler_rust/compiler/src/codegen/runtime_ffi.rs` — only the same
  `_with_expanded` pair (lines 333–334).
- `src/compiler_rust/runtime/src/value/` — no `rt_aes_sbox`, `rt_aes_rcon`, or
  `rt_aes128_encrypt_block_into` anywhere.

A reference implementation exists in an old worktree path
(`examples/simple_os/arch/shared/crypto_common.h`) but is not part of the active
runtime.

## Repro

Write any spec that calls `aes128_gcm_encrypt` or `aes128_gcm_decrypt` and run
`bin/simple test`. All specs fail immediately with:

```
semantic: unknown extern function: rt_aes_sbox
```

(or `rt_tls13_aes128_gcm_encrypt` for encrypt paths, which hits the fast-path
`rt_tls13_aes128_gcm_encrypt` call before reaching `rt_aes_sbox`.)

## Context

The most recent commit touching this file (`ab8d51ba42`) predates the file's
introduction commit (`332be1b80f`, message: "pick up OS/crypto WIP"). The WIP
label in that commit indicates this was a known incomplete landing — the source
was committed but the extern wiring was not.

AES-256-GCM (`src/os/crypto/aes_gcm.spl`) works correctly in interpreter mode
because it uses only `rt_aes_encrypt_block_with_expanded`, which is fully registered.

## Impact

NIST SP 800-38D Appendix B test vectors (TC1–TC4) for AES-128-GCM cannot be
verified until this is resolved. Agent BB's "AES-256 only" report (commit
`34bd3e1915`) was accurate in effect.

## Next Steps (not a proposed fix — pointing at the gap)

Either:

1. Wire the 4 missing externs in `src/compiler_rust/compiler/src/interpreter_extern/`
   and `src/compiler_rust/runtime/src/value/aes.rs` (using the existing AES tables
   already present in `aes.rs`), **then** add the NIST SP 800-38D Appendix B spec at
   `test/unit/lib/crypto/aes128_gcm_nist_vectors_spec.spl`; or

2. Rewrite `aes128_gcm.spl` to use `rt_aes_encrypt_block_with_expanded` (matching
   the AES-256 path), which would immediately unblock interpreter-mode execution.

Option 2 is the lower-friction unblock; Option 1 preserves the intended pure-table
design.
