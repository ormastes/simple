# Architecture: WM Glass Theme on Host and SimpleOS

## Decision

Use explicit resolved-package projection at target bootstrap. The file-backed
`ResolvedThemePackage` remains the only authority. Host startup projects it to
an immutable common-layer `ThemeRenderSnapshot`; bare metal imports the same
snapshot generated by the existing `theme-sync compile-to-spl` command and
verified against the package manifest. Render loops never read or parse theme
files.

This is composition plus adapters, not a new registry, MDSOC weave, private
renderer, or lazy auto-registration scheme.

## Layered Ownership

```text
config/themes/theme.sdn + aetheric package
  -> nogc_sync_mut.ui.theme_package (parse, validate, SHA-256 manifest)
  -> common.ui.theme_render_snapshot (immutable semantic/render value object)
     -> common.ui.wm_chrome_theme (WM projection)
     -> os.compositor.simple_web_window_renderer (package CSS + structure)
     -> common.ui.window_scene_draw_ir (semantic style projection)
        -> Engine2D (transient blur/shadow lowering and capability fallback)
```

- Package owner: alias resolution, CSS composition, typed material parsing,
  source manifest, semantic normalization, cache and invalidation.
- Common snapshot: theme ID/family/reference, source-manifest SHA-256,
  material SHA-256, CSS, colors, glass material, typography and fallback.
- WM adapter: maps the snapshot onto existing chrome state. Aqua literals are
  accessibility/emergency defaults only and are never a competing theme.
- Web adapter: appends structural layout CSS to snapshot CSS. It owns no visual
  palette.
- Scene/Draw IR: carries requested semantic style and theme revision only.
- Engine2D: owns transient blur/cache/backend material and reports realized
  capabilities; such state never enters Draw IR.

## Bootstrap and Invalidation

Hosted startup loads and validates `aetheric_dark`, constructs one snapshot,
installs WM projection, and creates compositor/Web caches with its fingerprint.
SimpleOS installs the generated snapshot before its first canonical desktop
frame. A host theme change loads once, atomically swaps the snapshot, increments
theme generation, marks scene/content dirty, and invalidates only theme-keyed
Web artifacts. No frame or input handler performs filesystem reads or scans.

The source fingerprint is SHA-256 over a canonical sorted manifest of
`path + content SHA-256`; the current `path:length` key is not an identity and
must be replaced. The normalized material SHA-256 excludes platform capability
results so host and SimpleOS compare requested semantics exactly.

## Bare-Metal Projection

`theme-sync compile-to-spl --theme=aetheric_dark` must stop emitting hard-coded
constants. It emits a tracked `ThemeRenderSnapshot` constructor containing the
composed CSS, typed material values, manifest hash and material hash. A drift
checker regenerates in memory and fails if the tracked snapshot differs. This
keeps `config/themes/**` authoritative without requiring QEMU VFS theme files.

## CSS and Render Repair

The existing Simple Web parser/cascade/layout path must retain RGBA, variables,
gradient, per-corner radius, borders, ordered shadow layers, backdrop
blur/saturation, typography and active/inactive selectors. Existing accessors
that return constant opacity/radius/shadow/blur/gradient values are defects,
not capability declarations. Unsupported realized effects select a named solid
fallback and emit capability evidence; they do not disappear silently.

Draw IR extends existing style properties/commands where necessary. Engine2D
must composite backdrop-dependent effects after prior scene content and before
surface/border/text. Host-GPU protocol changes, if needed, are versioned and
fail closed rather than accepting unknown style keys.

## Canonical Routes

- Host: `SharedWmScene -> DrawIrComposition -> Engine2D`; compatibility/direct
  rendering is rejected by feature evidence.
- SimpleOS: x86_64 `gui_entry_desktop.spl` -> `DesktopShell` ->
  `Engine2dWmFrameExecutor`; legacy `wm_entry.spl` is excluded.

## Pattern Evaluation

- Selected: explicit bootstrap adapter + immutable snapshot. It preserves
  dependency direction, removes hot-path I/O, and supports deterministic QEMU.
- Rejected: common importing `nogc_sync_mut` (layer cycle), renderer-local file
  loading (hot-path duplication), module-init auto-registration (unreliable on
  freestanding targets), runtime QEMU config reads (VFS/staging dependency),
  and a second CSS or WM renderer (parallel authority).
- MDSOC: no new virtual capsule or feature transform is needed. The package to
  snapshot compiler is a build-time projection, and runtime behavior remains
  ordinary next-layer adapters.

## Failure and Observability

Invalid packages, snapshot drift, unknown capabilities, fingerprint mismatch,
legacy/direct routes, stale evidence and missing accessibility fallback fail
before accepting a frame. Startup logs one structured theme record; each frame
reports route/fallback and theme generation without serializing full CSS.
Evidence records the highest GPU/readback proof rung and first unavailable rung.
