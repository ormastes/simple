## Scope

This research extends, rather than replaces, the earlier RISC-V documents under
`doc/01_research/hardware/riscv/` and the historical status report
`doc/09_report/riscv32_riscv64_fpga_simpleos_production_status_2026-07-03.md`.
It answers four narrower questions from the current request:

1. Is the RV32/RV64 CPU RTL actually generated by the Simple compiler?
2. Are Sv32, Sv39, and RISC-V physical memory protection connected to execution?
3. Does either generated RTL lane boot Linux to an interactive login and run `ls`?
4. What evidence would qualify Simple as an FPGA design language rather than a
   bundle-description language?

The audit was read-only. Current source, current files, and current connected
hardware were treated as authoritative; old build reports were treated as
historical evidence only.

## Executive Finding

The requested end state is not implemented.

- Simple contains real behavioral RV32 and RV64 CPU/MMU/SoC models and a real
  VHDL backend, but the production CPU models are not the sources compiled into
  the current FPGA Linux bundle.
- The bundle generator emits empty VHDL CPU architectures and testbenches that
  report PASS without executing instructions. Its debug sidecar honestly says
  `readiness=contract` and `linuxBootValidated=false`, but surrounding smoke
  wrappers currently promote the unconditional marker to PASS.
- Sv32 and Sv39 walkers are disconnected from instruction fetch and load/store.
  RV64 stores MMU state and flushes it for `SFENCE.VMA`, but never calls the
  translation path. RV32 does not carry privilege, supervisor CSR, or MMU state
  in its core at all.
- RISC-V PMP is represented only by boot-time CSR-write planning. Neither CPU
  RTL owns `pmpcfg`/`pmpaddr` state or enforces permissions on fetch/load/store.
- Current hardware inventory sees the KV260 FT4232H and its Xilinx serial
  ports, but the prior RV32/RV64 bitstreams and ELF artifacts are absent from
  the current build tree. Vivado is not on the current `PATH`; GHDL, QEMU,
  Yosys, SymbiYosys, Boolector, and a RISC-V cross compiler are present.
- Historical physical evidence proves early SimpleOS UART/ILA markers on the
  KV260. It does not prove OpenSBI, Linux, a login prompt, interactive input, or
  `ls` on either generated CPU.

## Execution Call-Chain Audit

### RV32

`core_combinational()` drives instruction address directly from `pc` and data
address directly from the ALU result. It imports only decode, ALU, register
file, simple LSU load formatting, and RVFI. `csr_s.spl` and `mmu_sv32.spl` are
exported libraries with no production callers. No focused test imports
`mmu_translate()`.

Consequences:

- Supervisor and user mode are not core state.
- Any SYSTEM instruction outside semihosting halts the core.
- SATP, SFENCE.VMA, SRET, delegation, page faults, and PMP cannot occur in the
  executing RV32 path.
- Instruction fetch and data access cannot observe Sv32 or PMP.

### RV64

`core64_init()` creates CSR and MMU state, and `core64_update()` flushes the TLB
for decoded `SFENCE.VMA`. However, `core64_combinational()` still drives raw
instruction and ALU addresses. Searches found no production calls to
`mmu64_set_satp()`, `mmu64_translate()`, `csr_s64_write()`, or `csr64_rw()`.
`lsu64_access()` returns the virtual address unchanged for both Bare and
non-Bare SATP modes. `core64_step()` only increments PC by four.

Implementation note (2026-07-18): this paragraph records the initial audit.
The production APIs are now `sv39_walker64_start/cycle`,
`memory64_start/cycle`, and `core64_cycle`; the weaker direct-RAM function was
renamed `mmu64_translate_ram_test_adapter`, and the identity LSU, raw core-port,
and PC-only step paths were removed. The clocked SoC now routes ROM, DRAM,
CLINT, PLIC, and UART requests. M execution and machine interrupt sampling are
now connected. LR/SC and AMO.W/D use translated physical reservations and the
single-master protected bus. Integer RV64C now expands through the same legal
32-bit decode/commit path while preserving 2-byte PC/link/fault semantics;
complete supervisor interrupt contexts, MPRV data privilege, and RV32 parity
remain open.

The walker uses the 32-bit-addressed `RamState`, while the RV64 SoC uses
`Ram64State`; the types cannot form a real RV64 page-table bus without an owner
boundary change.

## Generated RTL and Evidence Audit

The current generated call chain is:

`check-riscv-rtl-linux-smoke.shs`
→ `rtl_riscv{32,64}_linux_generated.shs`
→ `ghdl_generated_*_linux_handoff_runner.shs`
→ `generate_riscv_fpga_bundle.spl`
→ `write_generated_core_lane()`.

At the final step:

- RV64 `.spl` source is only a short proof-lane text record.
- `generated_rvfi_core_vhdl()` emits ports with an empty architecture and
  default-zero outputs.
- `runner_tb_vhdl()` instantiates that entity, then reports its requested PASS
  marker unconditionally at time zero.
- The Sv39, boot-info, firmware-jump, and Linux-handoff testbenches therefore
  do not distinguish a CPU from an empty entity.
- The formal harness and `.sby` files establish structural plumbing, not ISA,
  MMU, PMP, or liveness properties over real retired instructions.
- `compile_to_vhdl_module()` in the same module explicitly emits a VHDL stub.

The focused RV64 Sv39 unit spec reimplements bit arithmetic locally instead of
importing `mmu64_translate()`. The GHDL Sv39 system spec accepts tool absence or
runner code 2 via tautological PASS assertions. There is no corresponding
production Sv32 translation spec.

Therefore the current smoke marker means “bundle and testbench elaborated,”
not “the generated CPU executed Linux handoff code.”

## Linux Boot Gap

The existing profile correctly records the pre-kernel register contract, but a
Linux-capable soft CPU also needs a continuous executing chain:

1. reset/boot ROM;
2. M-mode firmware or OpenSBI with timer, IPI, console, and platform DT;
3. S-mode Linux entry with `a0=hartid`, `a1=dtb`, and `satp=0`;
4. M/S/U traps and interrupts;
5. atomics and the advertised RV32GC/RV64GC instruction surface;
6. Sv32/Sv39 translation, TLB fences, page faults, and PMP;
7. enough writable memory for kernel plus initramfs;
8. UART RX and TX for an interactive getty;
9. a root filesystem whose prompt accepts login and runs `ls`.

Current generated lanes stop before item 1 because the emitted CPU is empty.
Historical K26 lanes execute a small early-instruction subset from BRAM and
prove only a boot marker. The 512 KiB/BRAM-style path is not a Linux memory
system. The existing AXI-HP bridge targets the external VexRiscv-SMP flow and
is not connected to a Simple-generated CPU.

QEMU software boot and generated-RTL boot must remain separate evidence rows.
QEMU can validate OpenSBI, Linux, DT, initramfs, login scripting, and the
platform contract. It cannot prove that Simple-generated RTL implements the
CPU. The exact same firmware/kernel/rootfs must later run on cycle RTL and the
physical FPGA.

## Current Host and Board Observation

Observed on 2026-07-18:

- USB `0403:6011` FT4232H is present.
- Xilinx ML Carrier Card serial links for interface ports 1, 2, and 3 exist.
- GHDL, `qemu-system-riscv32`, `qemu-system-riscv64`, Yosys, SymbiYosys,
  Boolector, and `riscv64-linux-gnu-gcc` resolve.
- Vivado does not resolve from the current `PATH`.
- The July 3 RV32 bitstream, the named K26 RV64 product bitstream, and both FPGA
  ELFs are absent. One older generic KV260 bitstream remains under
  `build/build/xilinx_kv260/gateware/`; its May 21 timestamp and missing paired
  product artifacts make it historical, not current dual-arch evidence.
- No non-Xilinx PMOD UART adapter appears under `/dev/serial/by-id`.

ILA can observe output-only boot markers, but an interactive Linux terminal
requires a bidirectional transport. The plan must either attach a 3.3 V PMOD
UART adapter to the PL UART pins or implement and prove another board-origin
RX/TX path. A PS UART transcript is not softcore evidence unless the bridge
from the generated CPU is part of the tested design.

## Requirement Baseline Derived from Local Evidence

- LR-001: The authoritative RV32 and RV64 CPU/SoC RTL shall be compiler output
  rooted in tracked `.spl` hardware entrypoints; emitted-string or copied CPU
  RTL may be diagnostic/oracle material only.
- LR-002: Bundle generation shall fail if the compiler uses a stub/fallback,
  emits an empty architecture, or omits clocked execution and memory ports.
- LR-003: RV32 fetch/load/store shall traverse Sv32 then PMP; RV64 shall
  traverse Sv39 then PMP. Bare M-mode behavior shall remain explicit.
- LR-004: Both cores shall implement the advertised M/S/U CSR, trap,
  interrupt, atomic, and fence behavior needed by OpenSBI and Linux.
- LR-005: MMU/PMP tests shall call production functions and observe physical
  address, permissions, page/access fault, TLB invalidation, and protected
  region denial. Tests that reproduce constants are not coverage.
- LR-006: Formal proof shall bind actual retired instructions and memory
  effects; a port-count or empty harness is readiness only.
- LR-007: QEMU, RTL simulation, synthesis, and physical FPGA shall consume a
  pinned common OpenSBI/Linux/DT/rootfs artifact set.
- LR-008: RV32 and RV64 shall each reach `login:`, accept input, reach a shell,
  execute `ls /`, and retain the transcript plus artifact hashes.
- LR-009: Physical evidence shall include board identity, bitstream hash,
  timing/resource report, programming transcript, and board-origin terminal
  transcript for each architecture.
- LR-010: Simple FPGA-language qualification shall cover compile diagnostics,
  source maps, deterministic RTL, simulation, formal, synthesis, timing,
  programming, and live execution without a parallel handwritten CPU path.

## Research Conclusion

The shortest honest route reuses the existing behavioral cores, MMUs, SoC
models, VHDL backend, manifests, and board wrappers. It removes the parallel
empty/string-emitted CPU path and integrates one architecture capability at a
time through the real compiler. An external core such as VexRiscv can remain a
differential oracle, but cannot satisfy the final “Simple-generated CPU” row.

## 2026-07-18 Attached-Board Evidence

Read-only host enumeration now proves a live Xilinx FT4232H interface
(`0403:6011`) with product `ML_Carrier_Card` and serial
`XFL1OSWWFM2B`.  Interfaces 01, 02, and 03 enumerate as `/dev/ttyUSB1`,
`/dev/ttyUSB2`, and `/dev/ttyUSB3`; interface 00 is the non-TTY JTAG lane.
The current host exposes `openocd` but no `vivado`, `vitis`, or `xsct` on
`PATH`.  AMD's KV260 platform documentation also names its board-side hardware
as the ML carrier card, so KV260 remains the best-supported board identity, but
the programming gate must still record an authoritative device/JTAG scan
before treating that inference as final hardware identity.
