# IPC `-38` Diagnosis — Agent R Research Slice (2026-04-13)

## Symptom
```
[wm-service] Failed to create IPC port (error -38)
```
Observed on x86_64 live QEMU `x64-desktop-test` lane during desktop bring-up,
after `[launcher-ready]` and before `[desktop-e2e] desktop-ready`.

## Authoritative answer

1. **Primitive/kernel or library error?** Kernel-primitive level, but specifically
   the **baremetal C syscall dispatcher** — not the Simple-language kernel.
   SimpleOS reuses POSIX errno numbering; `-38` is `ENOSYS` (function not
   implemented). See `src/os/posix/errno.spl:37` and `src/os/libc/include/errno.h:52`.

2. **Which layer?** `examples/simple_os/arch/x86_64/boot/baremetal_stubs.c`
   function `userlib__syscall_raw__syscall`. Its switch handles case 20
   (IPC_SEND) and case 21 (IPC_RECV) but had **no case 22** (`SYS_IPC_CREATE_PORT`),
   so every call fell through to `default: return -38; /* ENOSYS */`
   (line 3480). Cases 23 (service-id SEND) and 24 (CONNECT) also missing.

3. **Hosted parity?** Hosted path works. `src/os/kernel/ipc/syscall.spl:653`
   `_handle_ipc_create_port` calls `ipc_mgr.create_port(current, "port")` and
   returns the positive port id. The in-process `IpcManager.create_port`
   (see `src/os/kernel/ipc/ipc.spl:134`) is fully implemented. Only baremetal lacks it.


Hosted path (not taken on baremetal) would route id=22 through
`src/os/kernel/ipc/syscall.spl:237` -> `_handle_ipc_create_port` (line 653)
-> `ipc_mgr.create_port` -> positive id.

## Secondary finding — SYS_IPC_SEND id inconsistency

- Kernel syscall.spl: `SYS_IPC_SEND = 20`
- Most service modules (`wm_service.spl:40`, `launcher.spl`, `vfs_service.spl`,
  `driver_supervisor.spl`, `device_registry.spl`, `netstack_service.spl`,
  `virtio_net_service.spl`, `pcimgr.spl`): `SYS_IPC_SEND = 23`

Services issue SEND on id 23; baremetal only had id 20. This would be the
**next** failure after create_port unblocks. The stub fix in this slice
adds case 23 -> `_ipc_send_handler` and case 24 -> pseudo-id to preempt
the cascade.

## Fix landed

`examples/simple_os/arch/x86_64/boot/baremetal_stubs.c` — added:
- `case 22` SYS_IPC_CREATE_PORT -> return 1 (pseudo port id; baremetal has
  a single implicit `_ipc_reply` shared by all services)
- `case 23` service-id SYS_IPC_SEND -> routes to `_ipc_send_handler`
- `case 24` SYS_IPC_CONNECT -> return 1 (same implicit port)

Rationale: baremetal already uses a single-slot `_ipc_reply` struct for
all IPC, so port multiplexing is fictional — returning a constant
non-zero id lets services pass the `port < 0` check and proceed.
